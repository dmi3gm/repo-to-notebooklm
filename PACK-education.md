Pack, описывающий **универсальные** методы, принципы и инструменты преподавания знаний. Применим к обучению ЛЮБОМУ предмету — от нулевых принципов до профессиональных навыков.

## 1. Что это

PACK-education — source-of-truth для предметной области **«Методика обучения человека»**. Описывает **абстрактные** методы, принципы и инструменты преподавания знаний. Применим к обучению ЛЮБОМУ предмету.

**Объект описания:** Взаимодействие «преподаватель ↔ аудитория» как инженерный объект.

**Ключевое правило:** Pack EDU содержит ТОЛЬКО абстрактные методы, модели и шаблоны. Конкретное содержание (заполненные ячейки, банки вопросов, конкретные программы) — downstream.

- [ ] Новая сущность имеет frontmatter (id, name, status, summary, evidence)
- [ ] Различение содержит тест проверки
- [ ] Метод содержит Inputs/Outputs/Roles + evidence strength

4. Если сущность о созидателе как личности → PACK-personal

## Scope

**Bounded context**: [01A-bounded-context.md](01-domain-contract/01A-bounded-context.md)

### What This Pack Covers

This pack captures knowledge about **education methodology** — how to teach humans of different ages and professions:

- Teaching methods by age group (pedagogy, andragogy, gerontagogy)
- Teaching methods by profession (engineering, management, research)
- Instructional design frameworks (Bloom's taxonomy, ADDIE, backwards design)
- Learning theories (constructivism, cognitivism, connectivism, experiential)
- Assessment methods (formative, summative, competency-based)
- Teaching roles (instructor, facilitator, tutor, mentor, instructional designer)
- Failure modes in teaching (lecture=learning, one-size-fits-all, knowledge≠skill)
- Adaptation strategies for diverse audiences

### What This Pack Does NOT Cover

| Out of Scope | Where It Belongs |
|--------------|------------------|
| Creator characteristics and states | PACK-personal (PD) |
| Development program formats (seminar, practicum) | PACK-MIM |
| Specific course content | Downstream (DS-*) |
| IT platform implementation | PACK-digital-platform (DP) |
| Schedules, groups, operations | Operational docs |
| Motivational/inspirational content | Downstream engagement |

### [EDU.D.001] Педагогика ≠ Андрагогика

**Definition**: Педагогика — методика обучения детей; андрагогика — методика обучения взрослых. Фундаментальное различие не в возрасте, а в источнике мотивации и роли опыта.

**Contrast with**: Единая «методика обучения» без возрастной дифференциации.

| Педагогика | vs. | Андрагогика |
|------------|-----|-------------|
| Учитель определяет что, когда, как | | Ученик — соавтор процесса |
| Мотивация преимущественно внешняя | | Мотивация из жизненных задач |
| Предыдущий опыт минимален | | Опыт — главный ресурс обучения |
| Ориентация на предмет | | Ориентация на проблему/задачу |
| Отложенное применение | | Немедленное применение |

**Why it matters**: Применение педагогических методов к взрослым (EDU.FM.005) — одна из главных ошибок. Взрослый с 15-летним опытом не может учиться как первоклассник: ему нужна связь с его реальностью.

**Test**: Если обучение строится вокруг «я расскажу, ты запомни» без учёта опыта аудитории — это педагогический подход к взрослым (ошибка).

**Related items**:
- Failure Mode: [EDU.FM.005](../05-failure-modes/EDU.FM.005-ignoring-prior-experience.md)
- Formalization: [EDU.FORM.003](../02-domain-entities/formalizations/EDU.FORM.003-age-groups.md)

**SoTA**: `current`
- Basis: Knowles (1980) Andragogy in Action; Merriam & Bierema (2014); мета-анализы подтверждают различие в эффективности методов по возрастам
- Revision criterion: Если исследования покажут, что возрастная дифференциация не влияет на результаты обучения

---

### [EDU.D.002] Преподавание ≠ Фасилитация

**Definition**: Преподавание (instruction) — передача структурированного знания от эксперта к ученику. Фасилитация (facilitation) — создание условий, в которых ученик самостоятельно приходит к пониманию.

**Contrast with**: «Учить» как единый глагол без различения подхода.

| Преподавание (Instruction) | vs. | Фасилитация (Facilitation) |
|---------------------------|-----|----------------------------|
| Teacher-centered | | Learner-centered |
| Эксперт передаёт знание | | Процессист создаёт условия |
| Лекция, демонстрация, объяснение | | Вопросы, задачи, дискуссия |
| Эффективно для нового предмета | | Эффективно для интеграции знаний |
| Структура задана преподавателем | | Структура возникает из взаимодействия |

**Why it matters**: Выбор между преподаванием и фасилитацией зависит от аудитории, предмета и цели. Ни один из подходов не «лучше» — ошибка в неадекватном применении.

**Test**: Кто говорит >70% времени? Если преподаватель — это instruction. Если ученики — это facilitation.

**Related items**:
- Roles: [EDU.R.001](../02-domain-entities/roles/EDU.R.001-instructor.md), [EDU.R.002](../02-domain-entities/roles/EDU.R.002-facilitator.md)

**SoTA**: `current`
- Revision criterion: Если появятся модели обучения, делающие это различение нерелевантным

---

### [EDU.D.003] Формативное ≠ Суммативное оценивание

**Definition**: Формативное оценивание — оценка в процессе обучения для улучшения процесса. Суммативное — оценка после обучения для фиксации результата.

**Contrast with**: «Оценивание» как единый процесс.

| Формативное | vs. | Суммативное |
|-------------|-----|-------------|
| Во время обучения | | После обучения |
| Цель: улучшить процесс | | Цель: зафиксировать результат |
| Обратная связь для ученика | | Решение (зачёт/незачёт) |
| Низкие ставки | | Высокие ставки |
| Диагностическое | | Сертифицирующее |

**Why it matters**: Без формативного оценивания преподаватель не знает, что ученики не понимают — до итогового экзамена. Без суммативного — нет подтверждения результата.

**Test**: Используется ли результат оценки для изменения процесса обучения (формативное) или для принятия решения о допуске/сертификации (суммативное)?

**Related items**:
- Failure Mode: [EDU.FM.003](../05-failure-modes/EDU.FM.003-testing-knowledge-not-skills.md)

**SoTA**: `current`
- Basis: Black & Wiliam (1998) Inside the Black Box; Hattie (2009) Visible Learning
- Revision criterion: Если появятся модели оценивания, объединяющие оба типа без потери функций

### [EDU.D.004] Знание ≠ Навык (в контексте обучения)

**Definition**: Знание — понимание концепции, способность объяснить. Навык — способность действовать, применять знание в практике. Для обучения важно: **разные типы требуют разных методов**.

**Contrast with**: «Он знает, значит умеет» (или наоборот).

| Знание (Knowledge) | vs. | Навык (Skill) |
|--------------------|-----|---------------|
| «Знаю как» (декларативное) | | «Могу сделать» (процедурное) |
| Проверяется объяснением | | Проверяется действием |
| Метод: лекция, чтение, объяснение | | Метод: практика, тренировка, feedback |
| Уровни 1-2 Блума (запоминание, понимание) | | Уровни 3-6 Блума (применение → создание) |

**Why it matters**: Обучение только через лекции создаёт иллюзию понимания. Обучение только через практику без теории — хрупкий навык без переноса. Нужен баланс.

**Test**: Может ли ученик выполнить задачу в новом контексте (не только в учебном)? Если нет — есть знание, но нет навыка.

**Related items**:
- Failure Mode: [EDU.FM.003](../05-failure-modes/EDU.FM.003-testing-knowledge-not-skills.md)
- Failure Mode: [EDU.FM.004](../05-failure-modes/EDU.FM.004-content-without-practice.md)
- Cross-pack: PD.D.014 (Knowledge vs Skill в контексте агента)

**SoTA**: `current`
- Revision criterion: Если когнитивная наука покажет, что знание и навык — не различные конструкты

### [EDU.D.005] Содержание ≠ Методика

**Definition**: Содержание (what to teach) — предметная область, материал. Методика (how to teach) — дидактический подход, способ преподавания.

**Contrast with**: Эксперт в предмете = хороший преподаватель.

| Содержание (Content) | vs. | Методика (Pedagogy) |
|----------------------|-----|---------------------|
| Знание предмета | | Знание обучения |
| Что преподавать | | Как преподавать |
| Определяется Pack | | Определяется EDU |
| Меняется по домену | | Переносимо между доменами |

**Why it matters**: Великий учёный может быть ужасным преподавателем. Методическое мастерство — отдельная компетенция. Объединение в Pack-education этих двух вещей — ошибка: содержание принадлежит предметному Pack.

**Test**: Можно ли применить тот же метод преподавания к другому предмету? Если да — это методика. Если нет — это содержание.

**SoTA**: `current`
- Basis: Shulman (1986) «Pedagogical Content Knowledge» — признаёт пересечение, но подтверждает различие
- Revision criterion: Если методика окажется полностью специфичной для каждого предмета

### [EDU.D.006] Мотивация внешняя ≠ внутренняя

**Definition**: Внешняя мотивация — действие ради награды или избежания наказания. Внутренняя — действие ради интереса, автономии, чувства компетентности.

**Contrast with**: «Мотивированный ученик» без различения источника.

| Внешняя мотивация | vs. | Внутренняя мотивация |
|-------------------|-----|---------------------|
| Оценки, баллы, рейтинги | | Интерес, любопытство, вызов |
| Краткосрочная | | Долгосрочная |
| Исчезает с наградой | | Самоподдерживается |
| Эффективна для рутины | | Эффективна для творчества |

**Why it matters**: Для дошкольников внешняя мотивация (игра, похвала) — основной двигатель. Для взрослых — внутренняя мотивация критична. Выбор метода мотивации зависит от возрастной группы.

**Test**: Продолжит ли ученик обучение, если убрать награду? Если да — мотивация внутренняя.

**SoTA**: `current`
- Basis: Deci & Ryan (1985) Self-Determination Theory
- Revision criterion: Если SDT будет опровергнута или существенно пересмотрена

---

### [EDU.D.007] Обучение ≠ Информирование

**Definition**: Обучение — процесс, приводящий к изменению поведения или способностей ученика. Информирование — передача данных без проверки усвоения и без изменения способностей.

**Contrast with**: «Я рассказал — значит научил».

| Обучение (Teaching) | vs. | Информирование (Informing) |
|--------------------|-----|---------------------------|
| Проверяется результатом | | Проверяется фактом передачи |
| Включает обратную связь | | Однонаправленно |
| Адаптируется под ученика | | Одинаково для всех |
| Цель: изменить способности | | Цель: передать данные |

**Why it matters**: Подавляющее большинство «обучения» в организациях — информирование (презентации, рассылки, видео без проверки). Различение позволяет отделить реальное обучение от его имитации.

**Test**: Может ли ученик после «обучения» сделать что-то, чего не мог раньше? Если нет — было информирование.

**Related items**:
- Failure Mode: [EDU.FM.001](../05-failure-modes/EDU.FM.001-lecture-equals-learning.md)
- Cross-pack: MIM.D.001 (Семинар ≠ Практикум — аналогичное различение на уровне форматов)

**SoTA**: `current`
- Revision criterion: Если появятся данные, что информирование без обратной связи так же эффективно, как активное обучение

---

### [EDU.D.008] Адаптация ≠ Упрощение

**Definition**: Адаптация — изменение метода обучения под аудиторию (другой подход, другой темп, другие примеры). Упрощение — снижение сложности содержания (убрать трудное, оставить простое).

**Contrast with**: «Для детей нужно упростить» (без различения что менять — метод или содержание).

| Адаптация | vs. | Упрощение |
|-----------|-----|-----------|
| Меняется метод | | Меняется содержание |
| Сложность сохраняется | | Сложность снижается |
| Больше scaffolding, другие примеры | | Убрать трудные темы |
| Уважает способности аудитории | | Может недооценивать аудиторию |

**Why it matters**: Ребёнок 5 лет способен понять системное мышление — но через игру, а не через лекцию. Это адаптация метода, а не упрощение содержания. Упрощение — последнее средство, адаптация — первое.

**Test**: Сохранилась ли глубина предмета при изменении способа преподавания? Если да — адаптация. Если нет — упрощение.

**SoTA**: `current`
- Basis: Vygotsky (1978) Zone of Proximal Development; Tomlinson (2001) Differentiated Instruction
- Revision criterion: Если исследования покажут, что адаптация метода не влияет на результаты — только содержание имеет значение

---

### [EDU.D.009] Стиль обучения ≠ Уровень предварительных знаний

**Definition**: «Стиль обучения» (visual/auditory/kinesthetic) — ненаучная классификация, не влияющая на результаты. Уровень предварительных знаний (prior knowledge level) — главный предиктор эффективности обучения.

**Contrast with**: «Каждому ученику — свой стиль» без учёта реального уровня знаний.

| Стиль обучения (Learning Styles) | vs. | Уровень предварительных знаний (Prior Knowledge) |
|----------------------------------|-----|--------------------------------------------------|
| Нет валидной эмпирической базы | | Сильнейший предиктор результатов (d=0.94, Hattie) |
| Категория: visual/auditory/kinesthetic | | Континуум: новичок → продвинутый → эксперт |
| Подстройка под стиль не улучшает результаты | | Подстройка под уровень — ключевой фактор |
| Debunked: 4 мета-анализа, d=0.04 | | Current: Sweller (2024) Expertise Reversal Effect |

**Why it matters**: Индустрия «стилей обучения» отвлекает ресурсы от действительно важной адаптации — под уровень знаний. Методы, эффективные для новичков (scaffolding, worked examples), могут МЕШАТЬ экспертам (expertise reversal effect, Sweller 2024).

**Test**: Если адаптация учитывает «визуал vs аудиал» — ложная адаптация (❌). Если учитывает «новичок vs эксперт» — правильная адаптация (✅).

**Evidence strength**: Strong (debunked). Pashler et al. (2008), Kirschner (2017), Husmann & O'Loughlin (2019), Rogowsky et al. (2020) — все 4 мета-анализа: подстройка под стиль не улучшает результатов (d=0.04).

**Related items**:
- Failure Mode: [EDU.FM.002](../05-failure-modes/EDU.FM.002-one-size-fits-all.md) — одна из ложных «адаптаций»
- Distinction: EDU.D.008 — Адаптация ≠ Упрощение (правильная адаптация — под уровень знаний)

**SoTA**: `current`
- Basis: Pashler et al. (2008) "Learning Styles: Concepts and Evidence"; Sweller (2024) Expertise Reversal Effect; Hattie (2023) prior knowledge d=0.94
- Revision criterion: Если появятся рандомизированные контролируемые исследования, подтверждающие эффект стилей обучения

---

### [EDU.D.010] Ближний перенос ≠ Дальний перенос

**Definition**: Ближний перенос (near transfer) — применение навыка в похожих контекстах. Дальний перенос (far transfer) — применение навыка в принципиально новых контекстах. Ближний перенос надёжен и обучаем. Дальний перенос — крайне ненадёжен.

**Contrast with**: «Научился думать → сможет думать в любой области» (иллюзия дальнего переноса).

| Ближний перенос (Near Transfer) | vs. | Дальний перенос (Far Transfer) |
|---------------------------------|-----|-------------------------------|
| Похожий контекст | | Принципиально новый контекст |
| Надёжно обучаем | | Почти не наблюдается в исследованиях |
| Пример: Excel → Google Sheets | | Пример: шахматы → стратегическое мышление |
| Effect size: значимый | | Effect size: d ≈ 0 (Sala & Gobet, 2017) |

**Why it matters**: Вера в дальний перенос — основа множества образовательных программ («шахматы развивают мышление», «латынь улучшает логику», «программирование учит думать вообще»). Мета-анализы показывают: это иллюзия. Обучение должно быть максимально близко к контексту применения.

**Test**: Улучшаются ли результаты в ДРУГОЙ области после обучения? Если нет — дальний перенос не работает. Если результаты улучшаются только в похожих задачах — это ближний перенос.

**Evidence strength**: Strong. Sala & Gobet (2017) мета-анализ когнитивного тренинга: d ≈ 0 для дальнего переноса. Barnett & Ceci (2002) таксономия переноса: чем дальше контекст — тем меньше эффект.

**Related items**:
- Distinction: EDU.D.004 — Знание ≠ Навык (навыки переносятся хуже знаний)
- Method: EDU.M.002 — PBL (работает на ближний перенос через реальные проблемы)

**SoTA**: `current`
- Basis: Barnett & Ceci (2002) taxonomy of transfer; Sala & Gobet (2017) meta-analysis; Thorndike (1901) original transfer studies
- Revision criterion: Если будут найдены надёжные методы обучения дальнему переносу с воспроизводимыми результатами

---

### [EDU.D.011] Обучение человека ≠ Обучение ИИ (но параллели)

**Definition**: Обучение человека и обучение ИИ (machine learning) — различные процессы с разными механизмами. Но структурные параллели между ними продуктивны для проектирования обучающих систем.

**Contrast with**: «ИИ учится как человек» (антропоморфизм) или «ИИ не имеет отношения к обучению» (изоляция).

| Человек | ↔ Параллель | ИИ |
|---------|-------------|-----|
| Кривая забывания (Ebbinghaus) | Потеря информации со временем | Catastrophic forgetting |
| Рабочая память (7±2 элементов) | Ограничение на одновременную обработку | Context window |
| Desirable difficulties (Bjork) | Оптимальная трудность для обучения | Regularization, dropout |
| Scaffolding (ZPD) | Постепенное усложнение задач | Curriculum learning |
| Экзокортекс (внешняя память) | Расширение через внешние ресурсы | RAG (Retrieval Augmented Generation) |
| Spaced repetition | Интервальное повторение | Rehearsal methods |
| Interleaving | Чередование тем | Mixed training data batches |

**Why it matters**: Параллели позволяют (1) проектировать ИИ-ассистентов для обучения с учётом когнитивных ограничений, (2) использовать метафоры ИИ для объяснения процессов обучения, (3) находить новые методы обучения по аналогии. Но: различия ФУНДАМЕНТАЛЬНЫ — у человека есть сознание, мотивация, эмоции, телесный опыт.

**Test**: Можно ли провести аналогию между человеческим и машинным процессом? Если да — полезная параллель для проектирования. Но переносить НАПРЯМУЮ (без адаптации) — ошибка.

**Evidence strength**: Moderate. Параллели описаны в литературе (Zador, 2019; Lake et al., 2017), но формальная таксономия параллелей — emerging, не established SoTA.

**Related items**:
- Method: EDU.M.001 — Scaffolding ↔ Curriculum Learning в ML
- Formalization: EDU.FORM.004 — Three-Level Method Hierarchy (мета-принципы универсальны для обоих)
- Cross-pack: DP.CONCEPT.001 (Цифровой двойник — ИИ-аугментация развития)

**SoTA**: `emerging`
- Basis: Zador (2019) "A critique of pure learning"; Lake et al. (2017) "Building machines that learn and think like people"; Bengio (2017) curriculum learning
- Revision criterion: Если нейронауки покажут, что аналогии между мозгом и нейросетями фундаментально ошибочны

---

### [EDU.D.012] Профиль ≠ Уровень

**Definition**: Профиль изучающего принципы — **вектор** глубин владения по каждому принципу (ZP.1-N, FPF.*). Уровень — **скаляр** (единственное число, агрегат). Профиль — полная картина; уровень — редукция, теряющая информацию.

**Contrast with**: «У этого ученика уровень 3» (без указания, по каким принципам 3, а по каким 1).

| Профиль (Profile) | vs. | Уровень (Level) |
|-------------------|-----|-----------------|
| Вектор: {ZP.1:3, ZP.2:1, ZP.3:4, ...} | | Скаляр: «уровень 2.7» |
| Сохраняет информацию о каждом принципе | | Теряет информацию о различиях |
| Позволяет определить bottleneck | | Не показывает, где слабое место |
| Персонализирует обучение | | «Один размер для всех уровня X» |
| Как CEFR с отдельными баллами per skill | | Как единый балл GPA |

**Why it matters**: Два ученика с «уровнем 3» могут иметь радикально разные профили: {3,3,3,3,3,3} vs {5,1,5,1,5,1}. Первому нужна breadth-стратегия, второму — bottleneck-first. Уровень скрывает эту разницу. Профиль — основа персонализации.

**Формализация:** EDU.FORM.006 (Модель профиля мышления) определяет формат профиля. EDU.WP.004 (Learning Progression Map) хранит профиль ученика. EDU.WP.005 (Diagnostic Assessment) измеряет профиль.

**Аналогия:** В CEFR есть общий уровень (B2), но также есть отдельные оценки по reading, writing, listening, speaking. Рекомендация: если writing=A2, а reading=C1 — фокус на writing. Наш профиль работает так же, но с N осями.

**Test**: Потеряется ли информация при свёртке? Если {3,1,4,2} → «2.5» — потеряна информация о bottleneck (ZP.2=1 блокирует рост). Профиль сохраняет, уровень теряет.

**Related items**:
- Formalization: EDU.FORM.006 — Модель профиля мышления (TPM)
- Role: EDU.R.006 — Диагност узких мест (работает с профилем, не с уровнем)
- Work Product: EDU.WP.005 — Diagnostic Assessment (выход = профиль)

**SoTA**: `current`
- Basis: Competency-Based Medical Education (AAMC, 2024) — vectorial proficiency tracking; CEFR multi-skill assessment; Hattie (2023) — differential diagnosis of learning gaps
- Revision criterion: Если окажется, что скалярная оценка так же эффективна для персонализации, как векторная

### [EDU.D.013] Методологическая работа ≠ Методическая работа

**Definition**: Методологическая работа — получение решения (исследование, поиск ответа, reasoning). Методическая работа — передача понимания (проектирование объяснения для конкретного читателя/слушателя).

**Contrast with**: Путаница «написал решение = объяснил решение».

| Методологическая работа | vs. | Методическая работа |
|------------------------|-----|---------------------|
| Получить решение | | Передать понимание |
| Процесс рассуждения (reasoning trace) | | Дизайн объяснения |
| Результат: «трасса мышления» | | Результат: «текст для читателя» |
| Черновик как внешняя память | | Эссе как компилированный результат |
| Для себя | | Для конкретной аудитории |

**Why it matters**: Трасса рассуждения (как автор получил решение) и итоговый текст (как автор передаёт решение) — разные артефакты с разными качествами. Попытка передать трассу вместо компилированного объяснения перегружает читателя. Попытка объяснять без предварительного рассуждения — поверхностна.

**Трёхшаговый процесс**: (1) методологическая работа → идеальная модель решения, (2) модель читателя → ограничения bounded observer, (3) компиляция → текст для конкретного читателя.

**Test**: Кто адресат текста — автор или читатель? Если текст помогает АВТОРУ думать → методологический (черновик). Если текст помогает ЧИТАТЕЛЮ понять → методический (объяснение).

**Related items**:
- Distinction: EDU.D.005 — Содержание ≠ Методика (содержание — ЧТО передаём; методика — КАК)
- Cross-pack: PD.D.015 (Worldview vs Description) — методологическая работа формирует мировоззрение, методическая — описание

**SoTA**: `current`
- Basis: Левенчук А.И. (2026), Position Paper про Long-CoT — различение трассы рассуждения и итогового текста
- Revision criterion: Если будет показано, что трасса рассуждения и объяснение оптимально совпадают

**Definition**: Преподаватель (Instructor) — роль, чья задача — передача структурированного знания. Преподаватель владеет предметом и методикой, выстраивает последовательность, объясняет, демонстрирует и проверяет понимание.

**Key activities**:

- Демонстрация приёмов и способов действия
- Руководство практикой (guided practice)
- Контроль понимания (формативное оценивание)

**When ineffective**: Аудитория с опытом, задачи на синтез и создание, индивидуальная траектория. Ошибка: «преподаю = учатся» (EDU.FM.001).

**Distinction from Facilitator (EDU.R.002)**: Преподаватель передаёт знание (teacher-centered), Фасилитатор создаёт условия для его самостоятельного обнаружения (learner-centered). См. EDU.D.002.

**Definition**: Фасилитатор (Facilitator) — роль, создающая условия для самостоятельного открытия и интеграции знаний. Не передаёт знание напрямую, а направляет процесс через вопросы, задачи, дискуссии и рефлексию.

**Key activities**:
- Постановка провокативных вопросов
- Организация групповой работы и дискуссий
- Модерация обсуждений (равное участие, фокус)
- Организация рефлексии над опытом

**When effective**: Аудитория с предварительным опытом, задачи на интеграцию знаний, развитие критического мышления. Уровни 4-6 Блума (анализ, оценка, создание).

**When ineffective**: Полностью новый предмет без базовых знаний у аудитории. Без базы нечего интегрировать — нужен сначала Instructor.

**Distinction from Instructor (EDU.R.001)**: Фасилитатор не передаёт ответы — он создаёт ситуацию, в которой ученик находит ответ сам. «Кто говорит >70% времени?» — если преподаватель, это instruction; если ученики — facilitation.

```yaml
updated: 2026-03-21
note: "Renamed from Тьютор (Tutor). Расширение: добавлены навигационные obligations (мотивация, карта развития, агентность). При необходимости чистый Тьютор выделяется как подроль."
related:
  - EDU.M.001 # Scaffolding
  - EDU.D.001 # Педагогика ≠ Андрагогика
  - EDU.D.008 # Адаптация ≠ Упрощение
```

**Definition**: Навигатор (Navigator) — роль индивидуального сопровождения ученика, объединяющая адаптацию подачи материала с смысловой навигацией по программе развития. Навигатор адаптирует темп, глубину, примеры и метод под конкретного человека, а также объясняет зачем изучать тему, показывает место на карте развития и развивает агентность.

**Key activities**:
- Диагностика текущего уровня и пробелов
- Адаптация темпа и глубины под ученика
- Подбор индивидуальных примеров и задач
- Постепенное снятие поддержки (scaffolding, EDU.M.001)
- Мониторинг прогресса и корректировка плана
- Объяснение ЗАЧЕМ изучать тему (связь с жизнью и целями ученика)
- Показ места на карте развития (ступень, системные уровни)
- Развитие агентности (ученик сам решает, не система за него)
- Калибровка мотивации (не «надо», а «хочу потому что вижу»)

**When effective**: Индивидуальное обучение, ликвидация пробелов, потеря мотивации, смена этапа, вопрос «зачем мне это?». Работа с учеником, сильно отличающимся от среднего уровня группы.

**When ineffective**: Масштабирование (1:1 не масштабируется), групповая динамика (навигатор работает индивидуально). Чисто техническая маршрутизация (→ R13 Проводник).

**Distinction from Instructor (EDU.R.001)**: Навигатор адаптируется под ученика (1:1), Instructor ведёт группу по единому плану. Навигатор подстраивает и метод, и мотивацию; Instructor применяет единый метод к группе.

**Distinction from Mentor (EDU.R.005)**: Навигатор работает с конкретной программой развития (months). Ментор — с долгосрочным развитием личности через личный пример (years). Навигатор — для MVP. Ментор — на будущее (живые наставники, долгосрочные отношения).

**Distinction from Проводник (R13)**: Проводник = технический FSM-маршрутизатор (tier gating, кнопки). Навигатор = смысловое сопровождение (зачем, куда, мотивация). Разные уровни: Проводник — UI/UX, Навигатор — мировоззрение.

### [EDU.R.005] Ментор (Mentor)

```yaml
id: EDU.R.005
name: Ментор (Mentor)
status: active
summary: Роль долгосрочного развития через личный пример, обратную связь и передачу неявного знания
created: 2026-02-19
related:
  - EDU.D.001 # Педагогика ≠ Андрагогика
  - EDU.D.004 # Знание ≠ Навык

```

**Definition**: Ментор (Mentor) — роль долгосрочного развития ученика через личный пример, обратную связь и передачу неявного (tacit) знания. Ментор сам прошёл путь, который проходит ученик.

**Key activities**:
- Передача неявного знания через истории и примеры из своего опыта
- Обратная связь на реальные ситуации ученика
- Расширение картины мира (exposure to new perspectives)
- Поддержка в кризисах и точках выбора
- Моделирование поведения (ученик видит, как ментор действует)

**When effective**: Долгосрочное профессиональное развитие, передача неявного знания (которое невозможно формализовать), формирование профессиональной идентичности, навигация в карьере.

**When ineffective**: Передача формализуемого знания (для этого есть Instructor), оперативная подготовка к экзамену (для этого — Navigator), проектирование программы (для этого — Instructional Designer).

**Distinction from Navigator (EDU.R.003)**: Ментор работает с целостным развитием ученика в долгосрочной перспективе (years). Навигатор работает с конкретной программой развития (months) — адаптация + смысловая навигация. Ментор опирается на свой личный опыт — Навигатор на методику и данные ЦД.

**Distinction from Instructor (EDU.R.001)**: Ментор не ведёт занятия и не следует учебному плану. Взаимодействие ситуативно — ментор реагирует на реальные ситуации ученика, а не на заранее спланированную программу.

### [EDU.R.006] Диагност узких мест (Bottleneck Diagnostician)

```yaml
id: EDU.R.006
name: Диагност узких мест (Bottleneck Diagnostician)
status: active
summary: Определяет самый слабый принцип в профиле ученика и рекомендует приоритет обучения. Проводит диагностический тест (аналог IELTS placement test) для определения начального профиля
created: 2026-02-21
related:
  - EDU.FORM.006 # Модель профиля мышления (TPM) — источник данных
  - EDU.FORM.007 # Конструктор программ — потребитель диагностики
  - EDU.WP.004   # Learning Progression Map — вход
  - EDU.WP.005   # Diagnostic Assessment — инструмент диагностики
  - EDU.MAP.001  # Principle-Depth Cells — can-do утверждения как рубрика
  - EDU.D.009    # Уровень знаний (не стиль) — правильная ось адаптации
  - EDU.D.012    # Профиль ≠ Уровень — принципиальное различение
```

**Definition**: Диагност узких мест (Bottleneck Diagnostician) — роль, определяющая, какой принцип в профиле ученика является самым слабым звеном и больше всего ограничивает дальнейший рост. Текущая версия: простая формула. Будущая: ИИ-усиленная диагностика с учётом межпринципных зависимостей.

**Формула приоритизации:**

```
priority(P) = gap(P) × impact(P)

gap(P) = target_depth(P) − current_depth(P)
impact(P) = число принципов и навыков, заблокированных низкой глубиной P
```

**Межпринципные зависимости (impact graph):**

| Принцип | Блокирует (при low depth) |
|---------|--------------------------|
| ZP.1 Аксиоматичность | ZP.5 (Байес требует аксиоматики), FPF.Trust (формализация требует аксиом) |
| ZP.2 Структура | ZP.3 (мультимасштабность = структура на разных уровнях), FPF.Holon, FPF.Γ |
| ZP.3 Многомасштабность | ZP.4 (оптимизация требует видения масштабов), FPF.Bounded Context |
| ZP.4 Оптимизация | FPF.NQD-CAL, FPF.E/E-LOG |
| ZP.5 Вероятность | FPF.Trust (F,G,R), ZP.4 (оптимизация при неопределённости) |
| ZP.6 Ограничения | ZP.4 (оптимизация с ограничениями), FPF.Γ (WLNK) |

**Диагностический тест (аналог IELTS для принципов):**

Диагност проводит адаптивный тест для определения начального профиля (EDU.WP.005). Принцип работы:

1. **Каждый принцип тестируется независимо** — как IELTS тестирует reading и speaking отдельно
2. **Адаптивность:** начинает с глубины 2 (Understand), при правильном ответе — глубже, при неправильном — проще. 3-5 вопросов на принцип
3. **Can-do как рубрика:** каждый вопрос проверяет can-do из ячейки матрицы (EDU.MAP.001)
4. **Результат = вектор:** `{ZP.1: 3, ZP.2: 1, ZP.3: 4, ...}` — НЕ «уровень 2.5»
5. **Формы:** бот-тест (15-30 мин), self-assessment (10-15 мин), очное интервью (30-60 мин), портфолио-ревью (1-2 ч)

**Key activities**:
- **Первичная диагностика:** проведение теста (EDU.WP.005), определение начального профиля
- Чтение профиля ученика из Learning Progression Map (EDU.WP.004)
- Вычисление gap и impact для каждого принципа
- Ранжирование принципов по priority score
- Рекомендация фокуса: какой принцип учить следующим
- Определение готовности к переходу breadth-first → bottleneck-first
- **Повторная диагностика:** периодический re-test для отслеживания прогресса

**When effective**: Стратегия bottleneck-first (17+ лет или min(ZP) ≥ 2). При наличии достаточных данных в профиле (≥3 витка пройдено).

**When ineffective**: Breadth-first стратегия (все принципы на одной глубине — нет bottleneck). Начало обучения (нет данных для диагностики).

**Distinction from Navigator (EDU.R.003)**: Диагност определяет ЧТО учить (приоритет). Навигатор определяет КАК учить и ЗАЧЕМ (метод, темп, адаптация, мотивация). Диагност — аналитическая роль, Навигатор — роль сопровождения.

**Distinction from Instructional Designer (EDU.R.004)**: ID проектирует программу целиком. Диагност работает с конкретным учеником — адаптирует программу под его профиль.

**Эволюция роли:**

| Версия | Реализация |
|--------|-----------|
| v1 (текущая) | Простая формула: `priority = gap × impact`. Статический граф зависимостей |
| v2 (планируемая) | Динамический граф: impact пересчитывается по данным обучения |
| v3 (будущее) | ИИ-модель: учитывает скорость обучения, метакогнитивную зрелость, мотивацию |

### [EDU.M.001] Scaffolding

```yaml
id: EDU.M.001
name: Scaffolding
status: active
summary: Метод постепенного снятия поддержки — от полного руководства к самостоятельности ученика
created: 2026-02-19
related:
  - EDU.R.003 # Навигатор
  - EDU.D.008 # Адаптация ≠ Упрощение
  - EDU.FORM.001 # Таксономия Блума
```

**Definition**: Scaffolding (строительные леса) — метод обучения, при котором преподаватель сначала предоставляет максимальную поддержку, затем постепенно снимает её по мере роста компетенции ученика. Леса убирают, когда здание стоит само.

**Inputs**: Диагностика текущего уровня ученика; определение целевого уровня (зона ближайшего развития).

**Outputs**: Ученик выполняет задачу самостоятельно (без поддержки).

**Roles**: EDU.R.003 (Navigator) — основной исполнитель; EDU.R.001 (Instructor) — при групповом scaffolding.

**Steps**:
1. Диагностика: определить, что ученик может сам, а что — нет (ZPD)
2. Моделирование: показать выполнение задачи полностью
3. Совместное выполнение: ученик делает с помощью преподавателя
4. Частичная поддержка: ученик делает сам, преподаватель подсказывает
5. Самостоятельность: ученик выполняет без поддержки
6. Верификация: убедиться, что навык перенесён в новый контекст

**SoTA**: Vygotsky (1978) Zone of Proximal Development; Wood, Bruner & Ross (1976).

```yaml
  - EDU.R.002 # Фасилитатор
  - EDU.D.004 # Знание ≠ Навык
  - EDU.D.007 # Обучение ≠ Информирование
```

**Definition**: Problem-Based Learning — метод, при котором обучение начинается с реальной (или приближённой к реальности) проблемы. Ученики самостоятельно исследуют проблему, формулируют гипотезы и ищут решения — теория приходит как ответ на возникший вопрос, а не предшествует ему.

**Inputs**: Реальная или аутентичная проблема; базовые знания у аудитории для начала исследования.

**Outputs**: Решение проблемы + освоенные знания и навыки, необходимые для решения.

**Roles**: EDU.R.002 (Facilitator) — направляет процесс, не даёт ответов.

**Steps**:
1. Предъявление проблемы (ill-structured, без единственного верного ответа)
2. Анализ: что мы знаем? что нужно узнать?
3. Самостоятельное исследование (индивидуальное или групповое)
4. Обсуждение находок, формулирование решения
5. Презентация и защита решения
6. Рефлексия: что узнали, какие методы сработали

**When effective**: Подготовка профессионалов (медицина, инженерия, менеджмент); аудитория с базовыми знаниями; цель — перенос знаний в практику.

**When ineffective**: Полностью новый предмет без базы; аудитория, ожидающая готовых ответов; жёсткие временные ограничения.

**SoTA**: Barrows & Tamblyn (1980); Hmelo-Silver (2004) мета-анализ.

```yaml

name: Case Method

summary: Обучение через анализ реальных ситуаций (кейсов) с выработкой решений и обсуждением альтернатив

  - EDU.R.001 # Преподаватель (при разборе)
```

**Definition**: Case Method — метод обучения через анализ реальных ситуаций (case studies). Ученик ставится в позицию принимающего решение: анализирует контекст, данные и ограничения, вырабатывает решение и сравнивает с решениями других.

**Inputs**: Описание реальной ситуации (кейс) с достаточным контекстом для анализа; вопросы для обсуждения.

**Outputs**: Решение по кейсу + навык анализа ситуации и принятия решений в условиях неопределённости.

**Roles**: EDU.R.002 (Facilitator) — ведёт обсуждение; EDU.R.001 (Instructor) — при разборе «правильного» решения (если оно известно).

**Steps**:
1. Индивидуальное чтение и анализ кейса
2. Работа в малых группах: обсуждение, выработка решения
3. Презентация решений группами
4. Общая дискуссия: сравнение подходов, аргументация
5. Обобщение преподавателем: принципы и паттерны
6. Рефлексия: что перенести в свою практику

**When effective**: Управленческое и бизнес-образование; обучение принятию решений; развитие аналитического мышления; подготовка к реальным ситуациям.

**When ineffective**: Обучение процедурным навыкам (хирургия, программирование); аудитория без опыта в контексте кейса.

**SoTA**: Harvard Business School case method (Christensen, 1987); Ellet (2007) The Case Study Handbook.

### [EDU.M.004] Experiential Learning (Kolb)

```yaml
id: EDU.M.004
name: Experiential Learning (Kolb)
status: active
summary: Цикл обучения через опыт — конкретный опыт, рефлексия, концептуализация, эксперимент
created: 2026-02-19
related:
  - EDU.D.004 # Знание ≠ Навык
  - EDU.D.007 # Обучение ≠ Информирование
  - EDU.FM.004 # Контент без практики
```

**Definition**: Experiential Learning — модель обучения через циклическое прохождение четырёх фаз: конкретный опыт, рефлексивное наблюдение, абстрактная концептуализация и активный эксперимент. Каждая фаза необходима; пропуск любой — неполное обучение.

**Inputs**: Ситуация для получения опыта (реальная или смоделированная); готовность ученика к рефлексии.

**Outputs**: Интегрированное знание + навык, проверенный экспериментом в новом контексте.

**Roles**: EDU.R.002 (Facilitator) — организует рефлексию и концептуализацию; EDU.R.001 (Instructor) — помогает в фазе абстрактной концептуализации.

**The Cycle**:
1. **Concrete Experience** (CE): ученик делает, переживает, участвует
2. **Reflective Observation** (RO): ученик осмысляет, что произошло, наблюдает паттерны
3. **Abstract Conceptualization** (AC): ученик формулирует принципы, модели, теории
4. **Active Experimentation** (AE): ученик применяет концепции в новой ситуации

**When effective**: Обучение навыкам (лидерство, коммуникация, управление); обучение взрослых (андрагогика — опора на опыт); профессиональная подготовка.

**When ineffective**: Чисто теоретические предметы; аудитория без возможности получить опыт (только читают/слушают); если пропущена фаза рефлексии — опыт не конвертируется в знание.

**SoTA**: Kolb (1984) Experiential Learning; Kolb & Kolb (2005) Learning Styles.

### [EDU.M.005] Backwards Design (UbD)

```yaml
id: EDU.M.005
name: Backwards Design (UbD)
status: active
summary: Проектирование обучения от желаемых результатов назад — цели, оценка, учебные активности
created: 2026-02-19
related:
  - EDU.R.004 # Instructional Designer
  - EDU.FORM.001 # Таксономия Блума
  - EDU.FORM.002 # Модель ADDIE
  - EDU.D.005 # Содержание ≠ Методика
```

**Definition**: Backwards Design (Understanding by Design, UbD) — метод проектирования обучения, начинающийся с определения желаемых результатов, затем проектирования оценки (как узнаем, что цель достигнута?), и только потом — планирования учебных активностей.

**Inputs**: Предметная область; целевая аудитория; требования к результатам обучения.

**Outputs**: Полный дизайн учебного опыта: цели -> оценка -> активности (в этом порядке).

**Roles**: EDU.R.004 (Instructional Designer) — основной исполнитель.

**Three Stages**:
1. **Identify Desired Results**: Чему ученик должен научиться? Формулировка через глаголы Блума (EDU.FORM.001): «ученик сможет [глагол] [объект]»
2. **Determine Assessment Evidence**: Как мы узнаем, что цель достигнута? Проектирование оценки ДО проектирования активностей
3. **Plan Learning Experiences**: Какие активности приведут к результату? Выбор методов (EDU.M.001-004) и последовательности

**Key principle**: Assessment before activities. Типичная ошибка — начинать с «какой материал дать?» вместо «что ученик должен уметь делать?».

**When effective**: Проектирование курсов, программ, семинаров; любая ситуация, где нужен повторяемый дизайн обучения.

**When ineffective**: Импровизационное обучение; однократная менторская беседа; ситуации, где цели обучения неизвестны заранее.

**SoTA**: Wiggins & McTighe (1998, 2005) Understanding by Design; широко принят в K-12 и высшем образовании.

```yaml
id: EDU.M.006
name: Retrieval Practice
status: active
summary: Укрепление памяти и понимания через активное припоминание вместо пассивного перечитывания (d=0.50)
created: 2026-02-19
related:
```

**Definition**: Retrieval Practice — метод, при котором ученик активно вспоминает материал (recall), а не перечитывает (recognition). Каждый акт успешного припоминания укрепляет память сильнее повторного чтения. Известен как "testing effect".

**Inputs**: Изученный материал; retrieval cues (вопросы, задания); обратная связь.

**Outputs**: Укреплённая долговременная память; выявленные пробелы; метакогнитивная калибровка.

**Paттерны**: Low-stakes quiz, think-pair-share, flashcards (Anki), brain dump, practice test.

**Three-Level placement (EDU.FORM.004)**: Level 1 (Meta-Principle) — работает для ВСЕХ аудиторий, предметов, форматов.

**SoTA**: Roediger & Karpicke (2006); Dunlosky et al. (2013); Adesope et al. (2017).

```yaml
name: Spaced Repetition

summary: Распределение повторений по увеличивающимся интервалам для долговременного запоминания (d=0.54)
```

**Паттерны**: SRS Software (Anki, SuperMemo), Spiral Curriculum, Distributed Practice, Interleaved Review.

**Three-Level placement (EDU.FORM.004)**: Level 1 (Meta-Principle) — работает для ВСЕХ аудиторий, предметов, форматов.

**Human ↔ AI parallel (EDU.D.011)**: Кривая забывания Эббингауза ↔ catastrophic forgetting в нейросетях. Spaced repetition ↔ rehearsal methods.

**SoTA**: Ebbinghaus (1885); Cepeda et al. (2006); Wozniak (1987) SM-2 algorithm.

# [EDU.FORM.001] Таксономия Блума (Bloom's Taxonomy)

## Definition

Таксономия Блума — иерархическая классификация когнитивных целей обучения. Используется для формулирования учебных целей, выбора методов и проектирования оценки. Версия: Anderson & Krathwohl (2001) — пересмотренная таксономия с глагольной формой.

## The 6 Cognitive Levels

| Level | Name (EN) | Name (RU) | Verb Examples | Assessment Examples |
|-------|-----------|-----------|---------------|---------------------|
| 1 | **Remember** | Запоминание | перечислить, назвать, определить, воспроизвести | Тест на множественный выбор, определения |
| 2 | **Understand** | Понимание | объяснить, пересказать, интерпретировать, привести пример | Пересказ своими словами, объяснение аналогией |
| 3 | **Apply** | Применение | использовать, решить, продемонстрировать, реализовать | Решение задачи по образцу, выполнение процедуры |
| 4 | **Analyze** | Анализ | сравнить, разобрать, классифицировать, выявить причину | Сравнительный анализ, поиск ошибок, декомпозиция |
| 5 | **Evaluate** | Оценка | оценить, обосновать, критиковать, выбрать и защитить | Рецензия, обоснование решения, экспертная оценка |
| 6 | **Create** | Создание | спроектировать, создать, разработать, предложить | Проект, эссе, оригинальное решение, прототип |

## Visual Hierarchy

```
6. Create      ← высший уровень: создать новое
5. Evaluate    ← оценить, обосновать выбор
4. Analyze     ← разобрать, найти структуру
3. Apply       ← использовать в новой ситуации
2. Understand  ← объяснить своими словами
1. Remember    ← вспомнить, воспроизвести
```

## Usage Rules

1. **Формулировка целей**: «После обучения ученик сможет [глагол уровня N] [объект]»
2. **Выбор метода**: Уровни 1-2 — лекция, чтение. Уровни 3-4 — практика, кейсы. Уровни 5-6 — проекты, дискуссии, исследования.
3. **Проектирование оценки**: Уровень оценки должен соответствовать уровню цели. Тест на запоминание не проверяет навык анализа.
4. **Прогрессия**: Обычно двигаемся снизу вверх, но можно начинать с проблемы (PBL, EDU.M.002) и достраивать нижние уровни по необходимости.

## Distinctions

| Часто путают | На самом деле |
|-------------|---------------|
| «Знает» = уровень 1-2 | Знание без применения — не навык (EDU.D.004) |
| Тест = проверка обучения | Тест обычно проверяет уровни 1-2, а цели — на 3-6 (EDU.FM.003) |
| Все цели — уровень 6 | Не всё требует уровня «создание»; иногда достаточно уровня 3 |

## SoTA

- **Original**: Bloom (1956) Taxonomy of Educational Objectives
- **Revised**: Anderson & Krathwohl (2001) A Taxonomy for Learning, Teaching, and Assessing
- **Status**: `current` — широко принята, является стандартом в инструктивном дизайне
- **Revision criterion**: Если когнитивная наука предложит более точную модель уровней мышления

# [EDU.FORM.002] Модель ADDIE

## Definition

ADDIE — пятифазная модель инструктивного дизайна (Instructional Systems Design). Описывает полный жизненный цикл создания обучения: от анализа потребностей до оценки результатов. Каждая фаза производит артефакт, который является входом для следующей.

## The 5 Phases

### Phase 1: Analysis (Анализ)

**Question**: Кого учим, зачем, в каких условиях?

| Что анализировать | Артефакт |
|-------------------|----------|
| Целевая аудитория (возраст, опыт, мотивация) | Профиль аудитории |
| Проблема / потребность (зачем обучение?) | Обоснование программы |
| Текущий уровень vs целевой уровень (gap) | Gap-анализ |
| Ограничения (время, бюджет, формат) | Список ограничений |
| Контекст применения (где ученик будет использовать) | Описание среды переноса |

**Output**: Техническое задание на обучение (learning requirements).

### Phase 2: Design (Проектирование)

**Question**: Чему учим и как проверяем?

| Что проектировать | Артефакт |
|-------------------|----------|
| Цели обучения (по таксономии Блума, EDU.FORM.001) | Список целей с глаголами |
| Оценка (Backwards Design: оценка ДО контента) | Описание оценочных заданий |
| Структура программы (модули, последовательность) | Учебный план |
| Выбор методов (EDU.M.001-005) под каждую цель | Карта «цель -> метод -> оценка» |
| Выбор ролей (EDU.R.001-005) для реализации | Распределение ролей |

**Output**: Детальный дизайн-документ программы обучения.

### Phase 3: Development (Разработка)

**Question**: Какие материалы нужны для реализации дизайна?

| Что разрабатывать | Артефакт |
|-------------------|----------|
| Учебные материалы (тексты, презентации, видео) | Готовые материалы |
| Задания и упражнения | Банк заданий |
| Оценочные инструменты (тесты, рубрики, чеклисты) | Готовые инструменты оценки |
| Инструкции для преподавателей | Методическое руководство |
| Пилотная версия для тестирования | Прототип программы |

**Output**: Полный комплект учебно-методических материалов.

### Phase 4: Implementation (Реализация)

**Question**: Как запустить обучение и обеспечить его проведение?

| Что делать | Артефакт |
|-----------|----------|
| Подготовка преподавателей (train-the-trainer) | Обученные преподаватели |
| Пилотный запуск с малой группой | Обратная связь от пилота |
| Полноценный запуск | Проведённое обучение |
| Сбор данных для оценки | Данные (анкеты, результаты тестов) |

**Output**: Проведённое обучение + данные для оценки.

### Phase 5: Evaluation (Оценка)

**Question**: Достигнуты ли цели? Что улучшить?

| Тип оценки | Что проверяем | Когда |
|-------------|--------------|-------|
| Формативная (EDU.D.003) | Процесс обучения: понимают ли ученики? | Во время Implementation |
| Суммативная (EDU.D.003) | Результат: достигнуты ли цели? | После Implementation |
| Kirkpatrick Level 1 | Реакция: понравилось ли ученикам? | Сразу после |
| Kirkpatrick Level 2 | Обучение: научились ли? | Тесты/задания |
| Kirkpatrick Level 3 | Поведение: применяют ли на практике? | Через 1-3 месяца |
| Kirkpatrick Level 4 | Результаты: изменился ли бизнес-результат? | Через 6-12 месяцев |

**Output**: Отчёт об эффективности + рекомендации для следующей итерации.

## Flow

```
Analysis → Design → Development → Implementation → Evaluation
    ↑                                                    │
    └────────────── feedback loop ───────────────────────┘
```

**Key principle**: ADDIE — не водопад. Evaluation на каждой фазе даёт обратную связь для корректировки. Современные практики (SAM, agile ID) ускоряют цикл, но сохраняют все 5 фаз.

## Common Mistakes

| Ошибка | Следствие |
|--------|-----------|
| Пропуск Analysis — сразу к материалам | Обучение не решает реальную проблему |
| Design без Backwards Design | Оценка не соответствует целям |
| Development без пилота | Ошибки обнаруживаются на полном запуске |
| Implementation без train-the-trainer | Преподаватели не владеют методом |
| Evaluation только Level 1 (анкеты) | «Понравилось» ≠ «научились» |

## SoTA

- **Origin**: Florida State University (1975), для армии США
- **Iterations**: SAM (Successive Approximation Model), Agile ID — ускоренные версии
- **Status**: `current` — базовая модель, от которой отталкиваются все современные подходы
- **Revision criterion**: Если появится модель, покрывающая все 5 фаз более эффективно

# [EDU.FORM.003] Возрастные группы обучения

## Definition

Формализация четырёх возрастных групп, существенно различающихся по когнитивным особенностям, источникам мотивации и эффективным методам обучения. Разделение определяет выбор методики — применение одного подхода ко всем группам является ошибкой (EDU.FM.002).

## The 4 Age Groups

### Group 1: Дошкольники (Preschoolers) — 3-6 лет

```yaml
age_range: 3-6
discipline: Дошкольная педагогика
key_principle: Обучение через игру
```

| Характеристика | Значение для преподавания |
|----------------|--------------------------|
| **Когнитивное развитие** | Пре-операциональная стадия (Piaget): конкретное мышление, анимизм, эгоцентризм |
| **Внимание** | 10-20 минут непрерывного; требуется частая смена активности |
| **Память** | Преимущественно образная; запоминание через повторение и эмоцию |
| **Мотивация** | Внешняя: игра, похвала, интерес, новизна. Внутренняя мотивация — зачаточная |
| **Роль опыта** | Минимальный — каждый опыт новый |
| **Социальность** | Параллельная игра -> кооперативная. Обучение через имитацию |

**Effective methods**:
- Игра (ведущая деятельность возраста)
- Сенсорное обучение (потрогать, увидеть, услышать)
- Сторителлинг (истории, сказки)
- Имитация (повторение за взрослым)
- Короткие активности (5-15 мин) с переключениями

**Ineffective methods**:
- Лекция (нет абстрактного мышления)
- Длительные задания (>20 мин без переключения)
- Абстрактные объяснения без конкретных примеров
- Самостоятельная работа без руководства

**Key roles**: EDU.R.001 (Instructor) — как проводник через игру; EDU.R.003 (Navigator) — для индивидуальной работы.

---

### Group 2: Школьники (Schoolchildren) — 7-17 лет

```yaml
age_range: 7-17
discipline: Педагогика (школьное образование)
key_principle: Постепенный переход от руководства к самостоятельности
```

| Характеристика | Значение для преподавания |
|----------------|--------------------------|
| **Когнитивное развитие** | От конкретных операций (7-11, Piaget) к формальным операциям (12+): абстрактное мышление появляется к 12-13 годам |
| **Внимание** | 20-45 минут (растёт с возрастом); формируется произвольное внимание |
| **Память** | Развивается логическая память; мнемонические стратегии с 10-11 лет |
| **Мотивация** | Смешанная: оценки (внешняя) + интерес к предмету (внутренняя). Социальная мотивация (одобрение сверстников) сильна в 12-17 |
| **Роль опыта** | Нарастает; к старшим классам — значимый фактор |
| **Социальность** | Критична. Групповая работа эффективна. Peer learning с 10+ лет |

**Effective methods**:
- Scaffolding (EDU.M.001) — постепенное снятие поддержки
- Проектная работа (с 10+ лет)
- Групповая работа и peer learning (с 10+ лет)
- Визуализация и структурирование (схемы, карты)
- Геймификация (особенно 7-12 лет)

**Ineffective methods**:
- Только лекции без взаимодействия
- Абстрактная теория без примеров (до 12 лет)
- Длительная самостоятельная работа без чекпоинтов (до 14 лет)
- Игнорирование социальной динамики (12-17 лет)

**Key roles**: EDU.R.001 (Instructor) — основная роль; EDU.R.003 (Navigator) — для отстающих и продвинутых; EDU.R.002 (Facilitator) — для старших (15+).

---

### Group 3: Взрослые (Adults) — 18-60 лет

```yaml
age_range: 18-60
discipline: Андрагогика
key_principle: Опора на опыт + немедленная применимость
```

| Характеристика | Значение для преподавания |
|----------------|--------------------------|
| **Когнитивное развитие** | Формальные операции; мета-когнитивные навыки; критическое мышление |
| **Внимание** | 45-90 минут; но требуется осмысленность (скучное — не удерживает) |
| **Память** | Связывание нового с существующим опытом; ассоциативная память |
| **Мотивация** | Преимущественно внутренняя: решение реальных задач, карьера, самореализация. Внешняя (сертификат) — вторична |
| **Роль опыта** | Критичный ресурс. Игнорирование опыта — оскорбление (EDU.FM.005) |
| **Социальность** | Обмен опытом между участниками может быть ценнее лекции преподавателя |

**Effective methods**:
- Problem-Based Learning (EDU.M.002) — обучение через реальные задачи
- Case Method (EDU.M.003) — анализ реальных ситуаций
- Experiential Learning (EDU.M.004) — цикл Колба
- Дискуссии и обмен опытом (peer learning)
- Проекты с немедленным применением в работе

**Ineffective methods**:
- Школьная педагогика (лекция + экзамен без связи с практикой)
- Обучение «впрок» (абстрактно, без связи с текущими задачами)
- Игнорирование опыта аудитории (EDU.FM.005)
- Отсутствие возможности задать вопрос и обсудить

**Key roles**: EDU.R.002 (Facilitator) — основная роль; EDU.R.005 (Mentor) — для долгосрочного развития; EDU.R.004 (Instructional Designer) — для проектирования программ.

**Knowles' 6 Principles of Andragogy**:
1. Need to know (зачем мне это?)
2. Self-concept (я взрослый, я решаю сам)
3. Role of experience (мой опыт — ресурс)
4. Readiness to learn (учусь, когда мне нужно)
5. Orientation to learning (ориентация на задачу, не на предмет)
6. Motivation (внутренняя > внешней)

---

### Group 4: Старшие (Seniors) — 60+ лет

```yaml
age_range: 60+
discipline: Геронтагогика
key_principle: Адаптация темпа + уважение к опыту + поддержание когнитивного здоровья
```

| Характеристика | Значение для преподавания |
|----------------|--------------------------|
| **Когнитивное развитие** | Кристаллизованный интеллект сохранён/растёт; флюидный интеллект снижается (скорость обработки нового) |
| **Внимание** | Сохранено для значимых задач; снижено для рутины; утомляемость выше |
| **Память** | Рабочая память снижается; долгосрочная и процедурная — сохранены; нужно больше повторений для нового |
| **Мотивация** | Внутренняя: смысл, вклад, передача опыта, когнитивное здоровье. Карьерная мотивация — минимальна |
| **Роль опыта** | Максимальная. Главный ресурс. Часто старший — эксперт, а не новичок |
| **Социальность** | Потребность в общении; обучение как социальная активность. Группа = поддержка |

**Effective methods**:
- Замедленный темп с увеличением повторений
- Связь нового с обширным существующим опытом
- Мультисенсорное представление (визуально + аудио + текст)
- Малые группы с дискуссией
- Практическое применение с немедленной обратной связью
- Регулярность (короткие частые занятия лучше редких длинных)

**Ineffective methods**:
- Быстрый темп без пауз и повторений
- Большие объёмы нового материала за раз
- Мелкий шрифт, тихий звук, сложный интерфейс
- Обучение «сверху вниз» без учёта их экспертизы
- Тесты и экзамены с жёсткими временными ограничениями

**Key roles**: EDU.R.001 (Instructor) — с адаптацией темпа; EDU.R.003 (Navigator) — для индивидуального сопровождения; EDU.R.002 (Facilitator) — для обмена опытом в группе.

## Comparison Table

| Dimension | Дошкольники (3-6) | Школьники (7-17) | Взрослые (18-60) | Старшие (60+) |
|-----------|-------------------|-------------------|------------------|---------------|
| Ведущий метод | Игра | Scaffolding | PBL / Case | Темп + повторение |
| Мотивация | Внешняя (игра) | Смешанная | Внутренняя (задача) | Внутренняя (смысл) |
| Роль опыта | Минимальна | Нарастает | Критична | Максимальна |
| Длительность занятия | 10-20 мин | 20-45 мин | 45-90 мин | 30-60 мин |
| Абстракция | Нет | С 12+ лет | Да | Да (но медленнее) |
| Основная роль | Instructor (игра) | Instructor | Facilitator | Instructor + Navigator |

## Expertise Reversal Effect (Cross-Age)

Критически важно: **не только возраст, но и уровень знаний** определяет
выбор метода (EDU.D.009). Sweller (2024): методы, эффективные для новичков
(worked examples, scaffolding), могут МЕШАТЬ экспертам — и наоборот.

Пример: ребёнок 12 лет, 5 лет занимающийся программированием — эксперт.
Взрослый 40 лет, впервые открывший Python — новичок. Возрастная группа —
необходимый, но НЕ достаточный критерий выбора метода.

**Правило:** Возрастная группа (FORM.003) + Уровень знаний (D.009) = выбор метода.

## SoTA

- **Pedagogy**: Piaget (1952), Vygotsky (1978)
- **Andragogy**: Knowles (1980) Andragogy in Action; Merriam & Bierema (2014)
- **Gerontagogy**: Findsen & Formosa (2011) Lifelong Learning in Later Life
- **Expertise Reversal**: Sweller (2024) — уровень знаний модифицирует эффект возрастной группы
- **Evidence strength**: Strong — возрастная дифференциация подтверждена мета-анализами
- **Status**: `current`
- **Revision criterion**: Если исследования покажут, что возрастная дифференциация не влияет на эффективность обучения

# [EDU.FORM.004] Three-Level Method Hierarchy

## Definition

Трёхуровневая иерархия методов обучения — формализация, упорядочивающая
все методы по степени универсальности. Каждый уровень имеет свою область
применимости и evidence base.

## The Three Levels

```
Level 1: Meta-Principles (когнитивная наука)
    │    Работают ВСЕГДА, для ВСЕХ аудиторий
    │    Evidence: strong (мета-анализы, replicated)
    ↓
Level 2: Domain Methods (педагогические архитектуры)
    │    Работают для ОПРЕДЕЛЁННЫХ аудиторий/предметов
    │    Evidence: moderate-to-strong (контекстно-зависимые)
    ↓
Level 3: Context Techniques (локальные приёмы)
    │    Работают В КОНКРЕТНОЙ ситуации
    │    Evidence: local (experience-based, не generalized)
```

## Level 1: Meta-Principles

Основаны на когнитивной науке. Работают для всех возрастов, предметов,
форматов. Не зависят от контекста.

| Принцип | Суть | Effect Size | Источник |
|---------|------|-------------|----------|
| **Retrieval Practice** | Вспоминание > перечитывание | d=0.50 | Adesope et al. (2017) |
| **Spaced Repetition** | Распределение > концентрация | d=0.54 | Cepeda et al. (2006) |
| **Interleaving** | Чередование тем > блочное изучение | g=0.42 | Brunmair & Richter (2019) |
| **Feedback** | Обратная связь ускоряет обучение | d=0.70 | Hattie & Timperley (2007) |
| **Cognitive Load Management** | Не перегружать рабочую память | — | Sweller et al. (2019) CLT |
| **Desirable Difficulties** | Оптимальная трудность > комфорт | — | Bjork & Bjork (2011) |
| **Elaboration** | Связывание нового с известным | d=0.75 | Dunlosky et al. (2013) |

**Правило:** Мета-принципы применяются ВСЕГДА, независимо от предмета
или аудитории. Если метод обучения нарушает мета-принцип — метод ошибочен.

## Level 2: Domain Methods

Педагогические архитектуры, эффективные для определённых аудиторий
и предметных областей.

| Метод | Лучше всего для | Хуже всего для | Ссылка |
|-------|-----------------|----------------|--------|
| **Scaffolding** | Новички, сложные задачи | Эксперты (expertise reversal) | EDU.M.001 |
| **PBL** | Прикладные области (медицина, инженерия) | Абстрактные начальные знания | EDU.M.002 |
| **Case Method** | Менеджмент, право, бизнес | Точные науки (мало «кейсов») | EDU.M.003 |
| **Experiential Learning** | Навыки взаимодействия, лидерство | Формальные знания (математика) | EDU.M.004 |
| **Backwards Design** | Проектирование любых курсов | — (универсален, но сложнее для непрофессионалов) | EDU.M.005 |
| **Direct Instruction** | Новички, базовые навыки | Продвинутые (нужна автономия) | (общеизвестен) |

**Правило:** Доменные методы должны подчиняться мета-принципам Level 1.
Если PBL не включает retrieval practice и feedback — он неполон.

## Level 3: Context Techniques

Локальные приёмы, эффективные в конкретной ситуации. Не генерализуются.

| Техника | Контекст | Почему не Level 2 |
|---------|----------|-------------------|
| Think-Pair-Share | Лекция, 30+ человек | Зависит от размера группы и культуры |
| Jigsaw | Семинар, 10-25 человек | Зависит от делимости материала |
| Muddiest Point | Конец занятия | Формативный feedback, локальный формат |
| One-Minute Paper | Конец лекции | Локальный retrieval practice |
| Gallery Walk | Проектная работа | Зависит от физического пространства |

**Правило:** Context techniques — реализации мета-принципов и доменных
методов в конкретных условиях. Think-Pair-Share = retrieval practice
(Level 1) + facilitation (Level 2) в формате лекции для 30 человек (Level 3).

## Usage Rules

1. **Сверху вниз**: сначала проверь мета-принципы, затем выбери доменный
   метод, затем подбери контекстные техники
2. **Не путай уровни**: "retrieval practice не работает для моей аудитории"
   — ошибка (Level 1 работает всегда; возможно, неверно реализован на Level 3)
3. **Adaptation на Level 2-3**: адаптация под аудиторию происходит
   на уровнях 2 и 3, не на уровне 1
4. **Evidence соответствует уровню**: не требуй мета-анализов для Level 3
   техник; не принимай anecdotal evidence для Level 1

## Distinctions

| Часто путают | На самом деле |
|-------------|---------------|
| «Retrieval practice не подходит для детей» | Level 1 работает для всех; нужна адаптация формата (Level 3) |
| «PBL — лучший метод» | PBL — Level 2 (не для всех), должен включать Level 1 принципы |
| «Think-Pair-Share — метод обучения» | Think-Pair-Share — Level 3 техника, реализующая Level 1+2 |

## Parallel with EDU.D.011 (Human ↔ AI)

| Level | Человек | ИИ |
|-------|---------|----|
| Meta-principles | Spaced repetition, retrieval | Rehearsal, regularization |
| Domain methods | PBL, scaffolding | Curriculum learning, fine-tuning |
| Context techniques | Think-pair-share | Batch size tuning, prompt engineering |

## SoTA

- **Status**: `emerging` — иерархия как формализация новая (EDU Pack),
  но компоненты (каждый уровень) — established SoTA
- **Basis**: Dunlosky et al. (2013) evidence rankings; Hattie (2023)
  Visible Learning; Sweller et al. (2019) CLT; Merrill (2002)
  First Principles of Instruction
- **Revision criterion**: Если обнаружится, что мета-принципы контекстно-
  зависимы (т.е. Level 1 не универсален)

# Формализация: Оценка совместимости с культурой обучения (Compatibility Assessment)

## 1. Определение

**Оценка совместимости** — структурированный self-assessment для определения готовности человека к серьёзной программе развития. Не «вступительный экзамен» (проверка знаний), а **фильтр ожиданий** (проверка совместимости установок). Несовместимые ожидания → разочарование → уход → негативный отзыв.

## 2. Зелёные флаги (Green Flags) — программа подходит

| # | Флаг | Индикатор |
|---|------|-----------|
| 1 | **Цель — «поумнеть» в целом** | Не «выучить конкретный навык за 2 недели» |
| 2 | **Готовность к мышлению письмом** | Согласие документировать мышление, не только «слушать и запоминать» |
| 3 | **Ценит критику и измерения** | Хочет обратную связь по существу, а не одобрение |
| 4 | **Готовность менять методы работы** | Даже если это противоречит привычкам |
| 5 | **Бесконечное развитие, не курс** | Ищет стиль жизни, не «диплом за 3 месяца» |
| 6 | **Опытный специалист ищет выход из колеи** | Уже профи, но чувствует стагнацию |
| 7 | **Разбирается с новым, что раньше давалось легче** | Замечает, что скорость обучения снизилась |

## 3. Красные флаги (Red Flags) — сейчас не подходит

| # | Флаг | Почему не подходит |
|---|------|--------------------|
| 1 | **Ищет быстрое решение конкретной проблемы** | Программа развития — не консалтинг и не пожарная команда |
| 2 | **Нужен государственный диплом** | Программы не выдают формальных документов — результат в мастерстве |
| 3 | **Относится к наставникам как к «продавцам книг»** | Наставник удерживает стандарт, не обслуживает клиента (EDU.FM.005) |
| 4 | **Интересует «национальная истина» вместо универсальных методов** | Программа основана на трансдисциплинарных методах |
| 5 | **Чрезвычайные обстоятельства** | Нет ресурсов для регулярной работы (PD.FAIL.022 — Fatal Strategic Error) |
| 6 | **Ожидание развлекательного обучения** | «Обучение должно быть весело» — отсутствие готовности к серьёзной аналитической работе |

## 4. Дерево решений (три уровня фильтрации)

```
Уровень 1: ЦЕЛЬ
├── Разовое обучение конкретному → НЕ программа развития (курс, тренинг)
└── Бесконечное развитие → Уровень 2

Уровень 2: ФОРМАТ
├── Не готов к мышлению письмом → Пока не подходит (начать с личного развития)
└── Готов фиксировать мышление → Уровень 3

Уровень 3: ВЫБОР ПРОГРАММЫ
├── Нужно восстановить базу (сон, ритм, внимание) → Личное развитие (MIM.PRG.001)
├── Нужно повысить мастерство в работе → Рабочее развитие (MIM.PRG.002)
└── Интересует фронтир методов → Исследовательское развитие (MIM.PRG.003)

ФИНАЛЬНЫЙ ФИЛЬТР (для любой программы):
├── Согласен: развитие измеряется делом, не впечатлениями? → ДА → входит
├── Согласен: требуется ~10 ч/нед регулярной работы? → ДА → входит
├── Согласен: наставник удерживает стандарт, не обслуживает? → ДА → входит
└── Любой НЕТ → Обсудить ожидания, возможно — пока рано
```

## 5. Важная коррекция: развитие ≠ жертва

> «Не ясно зачем отказываться от удовольствия в выходные, как будто развитие — это что-то ужасное и неприятное. Это как раз неправильный мем, который выправляем в числе первых.»

При представлении требований (~10 ч/нед) критично **не создавать** фрейминг жертвы:
- НЕ: «2 часа в день на 5-дневке, откажитесь от выходных»
- ДА: «Ежедневное удержание внимания на развитии, в сумме 10 часов за неделю — это стиль жизни, не требующий больших ресурсов»

См. PD.FAIL.044 (Development-as-Sacrifice Framing) и PD.D.100 (Development as Lifestyle vs Course).

# EDU.FORM.006 — Модель профиля мышления (МПМ)

## Назначение

Модель для оценки, планирования и персонализации обучения принципам мышления. Заменяет скалярную шкалу («уровень 3 из 6») на **векторный профиль**: каждый принцип — отдельная ось, глубина по каждой оси — от 0 до 5 (шкала Bloom).

## Три фундаментальных решения

### 1. Список принципов открыт

Текущие принципы в программе:

| Группа | Принципы | Примечание |
|--------|----------|-----------|
| **Нулевые (ZP)** | ZP.1 Аксиоматичность, ZP.2 Структура, ZP.3 Многомасштабность, ZP.4 Оптимизация, ZP.5 Вероятность, ZP.6 Ограничения | Ядро; могут быть добавлены новые |
| **Первые (FPF)** | Holon, Bounded Context, Role/Capability, Trust (F,G,R), Γ-algebra, NQD-CAL, E/E-LOG | Вводятся по мере глубины |

Список может расширяться при обнаружении новых транс-дисциплинарных мета-ограничений (для ZP) или первых принципов (для FPF). Архитектура профиля допускает добавление осей без перестройки модели.

### 2. Глубина = Bloom (0-5), без именованных уровней

| Глубина | Bloom | Что может делать | Пример (ZP.1 Аксиоматичность) |
|---------|-------|-----------------|-------------------------------|
| **0** | — (не введён) | Не знаком с принципом | Не задаёт вопрос «из чего это следует?» |
| **1** | Remember | Узнаёт, может назвать | «Это когда проверяют, откуда взялось утверждение» |
| **2** | Understand | Объясняет своими словами | «Любое утверждение должно быть обосновано — цепочкой до аксиом» |
| **3** | Apply | Применяет к новой ситуации | Проверяет новостной источник: «из какой посылки следует этот вывод?» |
| **4** | Analyze | Выявляет нарушения, разбирает | Находит скрытую посылку в аргументе, определяет точку расхождения |
| **5** | Create | Создаёт системы на принципе | Строит формальную теорию от аксиом, проектирует курс обучения аксиоматичности |

Нет именованных уровней («Наблюдатель», «Мастер»). Глубина — число от 0 до 5. Это устраняет коллизии с другими терминами и делает шкалу универсальной.

### 3. Профиль = вектор, а не скаляр

Профиль человека — набор пар (принцип, глубина):

```yaml
profile:
  ZP.1_axiomaticity: 3      # Apply
  ZP.2_structure: 2          # Understand
  ZP.3_multiscale: 4         # Analyze
  ZP.4_optimization: 1       # Remember
  ZP.5_probability: 3        # Apply
  ZP.6_limits: 2             # Understand
  FPF.holon: 2               # Understand
  FPF.bounded_context: 1     # Remember
  FPF.trust: 0               # Не введён
  FPF.gamma: 0               # Не введён
```

**Агрегация:** `min(вектор)` показывает потолок (самый слабый принцип ограничивает целое). `mean(вектор)` показывает среднее. Оба полезны, но приоритет — min (bottleneck).

## Две стратегии обучения

### Стратегия 1: Breadth-First (дети, до ~17 лет)

**Принцип:** Все принципы проходятся параллельно, на каждом витке — все ZP на текущей глубине. FPF вводятся постепенно (начиная с глубины 2).

**Логика:** У детей нет базы для определения «узкого горлышка». Нужно равномерно формировать фундамент.

**Порядок:**
1. Глубина 1 по всем ZP (все 6 параллельно)
2. Глубина 2 по всем ZP + ввод FPF.holon, FPF.bounded_context
3. Глубина 3 по всем ZP + ввод FPF.role, FPF.trust
4. Глубина 4+ — переход к bottleneck-first (если готов)

**Когнитивные ограничения формы (не содержания!):**

| Примерный возраст | Когнитивные ограничения формы |
|-------------------|------------------------------|
| 3-6 | Игра, наблюдение, манипуляция. Сессия до 10 мин. Один принцип за раз |
| 7-10 | Эксперимент, проект, обсуждение. Сессия до 20 мин. 2-3 принципа за занятие |
| 11-14 | Анализ, дебаты, проектная работа. Сессия до 45 мин. Кросс-принципные связи |
| 14-17 | Исследование, формальная аргументация, кейсы. Сессия до 90 мин. Полный перенос |

> **Важно:** Таблица определяет ФОРМУ, а не СОДЕРЖАНИЕ. Ребёнок 8 лет на глубине 3 получает ту же задачу (Apply), что и взрослый на глубине 3 — но в другой форме (20-минутный эксперимент vs 90-минутный кейс-анализ).

### Стратегия 2: Bottleneck-First (17+ лет или по готовности)

**Принцип:** Определить самый слабый принцип в профиле → направить усилия на него.

**Формула приоритизации:**

```
priority(P) = gap(P) × impact(P)
```

Где:
- `gap(P)` = target_depth(P) − current_depth(P)
- `impact(P)` = число других принципов и навыков, заблокированных низкой глубиной P

**Пример:** Если ZP.1 (аксиоматичность) на глубине 1, а ZP.5 (вероятность) требует понимания аксиоматики для Байесовского вывода → impact(ZP.1) высокий → приоритет.

**Определение bottleneck:** Роль EDU.R.006 (Диагност узких мест).

**Переход с breadth-first на bottleneck-first:**
- Обязательный: когда min(профиль по ZP) ≥ 2 (все ZP хотя бы на Understand)
- Рекомендуемый: 17+ лет (достаточная метакогниция для самодиагностики)
- Возможный раньше: при явной метакогнитивной зрелости (по оценке EDU.R.006)

## Определение методики

### Методика = f(принцип, глубина)

Для каждой пары (принцип, глубина) существует рекомендуемый набор методов:

| Глубина | Bloom | Методы (EDU.M.*) |
|---------|-------|------------------|
| 1 (Remember) | Scaffolding (M.001), Retrieval Practice (M.006) | Ввод через наблюдение, примеры, повторение |
| 2 (Understand) | Scaffolding (M.001), Spaced Repetition (M.007), PBL начальный (M.002) | Объяснение своими словами, аналогии |
| 3 (Apply) | PBL (M.002), Experiential (M.004), Backwards Design (M.005) | Решение задач в новом контексте |
| 4 (Analyze) | Case Method (M.003), PBL продвинутый (M.002) | Поиск нарушений, декомпозиция, критика |
| 5 (Create) | Experiential (M.004), Spiral Curriculum (M.008), менторство | Проектирование, создание, обучение других |

**Зависимость от принципа:** Некоторые принципы имеют специфику:
- ZP.2 (Структура) — на глубине 2-3 эффективен CPA (Concrete→Pictorial→Abstract)
- ZP.5 (Вероятность) — на глубине 3+ требует работы с числовыми данными
- FPF.Trust — не раньше глубины 3 (требует понимания формализации)
- FPF.Γ-algebra — не раньше глубины 4 (требует формального мышления)

### Форма = f(когнитивные ограничения)

Форма подачи определяется когнитивными возможностями (связанными с возрастом, но не равными ему):

| Когнитивный уровень | Признаки | Доступные формы |
|---------------------|---------|----------------|
| Дооперациональный | Эгоцентризм, отсутствие сохранения, синкретизм | Игра, манипуляция предметами, визуальное, 5-10 мин |
| Конкретно-операциональный | Сохранение, обратимость, классификация | Эксперимент, CPA, проекты с руками, 15-25 мин |
| Формально-операциональный | Гипотетико-дедуктивное, абстрактное | Дебаты, формальная аргументация, кейсы, 30-90 мин |
| Постформальный | Диалектическое, системное, метакогниция | Исследование, проектирование систем, менторство, без ограничения |

> **Разделение:** ЧТО учить = профиль (принцип × глубина). КАК учить = метод (зависит от принципа и глубины). В КАКОЙ ФОРМЕ = когнитивные ограничения (связаны с развитием, не с возрастом).

## Переход между глубинами (Mastery Gate)

Для перехода с глубины N на глубину N+1 по принципу P необходимо:

1. **Can-do утверждения:** Демонстрация ВСЕХ can-do для текущей глубины данного принципа
2. **Перенос:** Применение принципа в НОВОМ контексте (не в том, где учился)
3. **Retrieval:** Устойчивое припоминание через 1+ неделю после обучения

Can-do утверждения специфичны для каждой пары (принцип, глубина). Они описаны в Learning Progression Map (EDU.WP.004) и генерируются Конструктором (EDU.FORM.007).

## Параллельный трек (родитель + ребёнок)

Родитель и ребёнок проходят ОДИН принцип на РАЗНОЙ глубине:

| Ребёнок | Родитель | Совместная активность |
|---------|---------|----------------------|
| Глубина 1 | Глубина 2-3 | Наблюдение → обсуждение «а почему?» |
| Глубина 2 | Глубина 3-4 | Совместный эксперимент + объяснение |
| Глубина 3 | Глубина 4-5 | Совместный проект + критический разбор |

Правило: глубина родителя ≥ глубина ребёнка + 1. Если родитель на той же глубине — сначала работает над своей.

# EDU.FORM.007 — Конструктор программ (КП)

## Назначение

Генерирует конкретные учебные программы для обучения принципам мышления (ZP + FPF). Читает профиль ученика (EDU.FORM.006) → определяет следующий шаг → генерирует учебную единицу с методом, формой и bridge к предыдущему витку.

## Архитектура конструктора

### Четыре измерения программы

```
Учебная единица = Принцип × Глубина × Метод × Форма
```

| Измерение | Значения | Источник |
|-----------|---------|---------|
| **Принцип** | ZP.1-ZP.N, FPF.* (открытый список) | ZP repo, FPF spec |
| **Глубина** | 0-5 (Bloom: Remember → Create) | EDU.FORM.006 |
| **Метод** | EDU.M.001-008+ (зависит от принципа и глубины) | PACK-education methods |
| **Форма** | Микро (5-10 мин), Совместный (15-30 мин), Проект (1-2 нед), Исследование (4+ нед) | Когнитивные ограничения ученика |

### Генерация учебной единицы

Каждая **учебная единица** (lesson unit) = пересечение одного принципа с одной глубиной:

```yaml
unit:
  principle: ZP.1              # Какой принцип
```

## Правила генерации

### R1: Спиральность (каждый принцип на каждой глубине)

Каждый принцип проходится на КАЖДОЙ глубине от 1 до 5. Для текущих 6 ZP: **30 ZP-ячеек** минимум (6 × 5). FPF-ячейки добавляются начиная с глубины, определённой правилом ввода.

### R2: Преемственность (bridge между витками)

Каждая единица ОБЯЗАНА содержать `bridge` — явную связь с предыдущим витком того же принципа: «Помнишь, как мы…? Теперь…». Без bridge — ученик не видит преемственности.

### R3: Mastery-gate (переход по готовности)

Переход на следующую глубину — только после демонстрации can-do утверждений текущей глубины. Не по возрасту, не по времени — по готовности.

### R4: Meta-principles always apply (EDU.FORM.004)

Retrieval practice (EDU.M.006) и Spaced repetition (EDU.M.007) встроены в КАЖДУЮ единицу на КАЖДОЙ глубине. Начало каждого занятия = 2-минутный recall предыдущего.

### R5: Параллельный трек (parent + child)

Для каждой единицы ребёнка генерируется параллельная единица для родителя (на 1+ глубину выше) + совместная активность. Правило: depth(parent) ≥ depth(child) + 1.

### R6: Доменная ротация

Принципы преподаются через РАЗНЫЕ домены на разных витках. Запрещено использовать один домен для всех глубин одного принципа. Это обеспечивает перенос.

| Глубина | Примеры доменов |
|---------|----------------|
| 1 | Природа, быт, игра |
| 2 | Природа, математика, социальное |
| 3 | Физика, биология, общество, медиа |
| 4 | Наука, инженерия, бизнес, медиа |
| 5 | Междисциплинарное, проектирование, исследование |

### R7: Выбор следующего принципа (стратегия)

| Стратегия | Правило выбора |
|-----------|---------------|
| **Breadth-first** | Round-robin по всем принципам на текущей глубине. Все ZP на глубине N → все ZP на N+1 |
| **Bottleneck-first** | EDU.R.006 определяет принцип с max(gap × impact) → фокус на нём |
| **Смешанная** | Bottleneck-first для ZP + breadth-first для FPF (FPF вводятся последовательно) |

### R8: Правила ввода FPF-концепций

FPF не вводятся с глубины 1. Каждый FPF-концепт имеет min_depth для ввода:

| FPF Concept | Min depth (ZP) | Причина |
|-------------|----------------|---------|
| Holon | 2 | Требует понимания «часть-целое» |
| Bounded Context | 2 | Требует понимания контекстуальности |
| Role/Capability | 3 | Требует способности к абстрактной декомпозиции |
| Trust (F,G,R) | 3 | Требует формализации и оценки |
| Γ-algebra | 4 | Требует формального мышления |
| NQD-CAL | 4 | Требует формального мышления |
| E/E-LOG | 4 | Требует метакогниции + формализации |

> `min_depth` означает: ученик может начать FPF-концепт, когда mean(ZP глубины) ≥ min_depth.

## Типы программ (шаблоны)

### Тип 1: Годовая семейная программа (Family Year)

```
Длительность: 1 учебный год (36 недель)
Формат: 2 микро-урока/нед + 1 совместное занятие/нед
Стратегия: breadth-first
Цель: +1 глубина по всем ZP за год
Итого: 72 микро-урока + 36 совместных = 108 единиц
Покрытие: все ZP × 12 единиц на принцип + FPF-единицы по правилу R8
```

### Тип 2: Интенсив (Intensive)

```
Длительность: 4 недели
Формат: 1 занятие/день (15-30 мин)
Стратегия: breadth-first или bottleneck-first
Цель: +1 глубина по целевым принципам
Итого: 20 единиц (будни)
Покрытие: 6 ZP × 3 единицы + 2 FPF
```

### Тип 3: Мастерская (Workshop)

```
Длительность: 1 день (4-6 часов)
Формат: групповое очное занятие
Стратегия: один принцип, экспресс-спираль от глубины 1 до 3-4
Итого: 6-8 активностей
```

### Тип 4: Bottleneck Sprint

```
Длительность: 2-4 недели
Формат: ежедневные 15-30 мин фокусные занятия
Стратегия: bottleneck-first (EDU.R.006 определяет фокус)
Цель: +1-2 глубины по одному принципу-bottleneck
Итого: 10-20 единиц на 1 принцип
Покрытие: 1 ZP/FPF, интенсивно, с retrieval по остальным
```

### Тип 5: IWE-программа (для глубины 4-5)

```
Длительность: 12-16 недель
Формат: 2 онлайн-сессии/нед + самостоятельная работа + менторство
Стратегия: bottleneck-first + FPF-deep
Покрытие: ZP+FPF review + PKM + Экзокортекс + Domain Pack + исследование
Артефакт: работающий экзокортекс + первый Pack
Связь: шаблон экзокортекса (WP-40)
```

## Алгоритм генерации следующей единицы

```
1. Прочитать профиль ученика (EDU.FORM.006 → EDU.WP.004)
2. Определить стратегию:
   IF min(ZP depths) < 2 OR cognitive_level < formal_operational:
     strategy = breadth-first
   ELSE:
     strategy = bottleneck-first
3. Выбрать принцип:
   IF breadth-first:
     principle = next in round-robin at current depth
   ELSE:
     principle = EDU.R.006.diagnose(profile).top_priority
4. Определить target_depth = current_depth(principle) + 1
5. Выбрать метод: method_table[principle][target_depth]
6. Определить форму: form_table[cognitive_level]
7. Выбрать домен: НЕ тот, что был на предыдущей глубине (R6)
8. Сформулировать bridge: «Помнишь, как мы [предыдущий виток]? Теперь…»
9. Сгенерировать задание (Backwards Design: цель → оценка → активность)
10. Встроить retrieval practice: recall по 1-2 другим принципам
11. Если есть parent_track: сгенерировать параллельную единицу
12. Записать can-do утверждения для оценки
13. Обновить Learning Progression Map после выполнения
```

## Ограничения и предупреждения

1. **Конструктор не заменяет родителя/наставника.** Совместное занятие требует реального присутствия и внимания.
2. **Глубина не привязана к возрасту.** Взрослый может быть на глубине 1, ребёнок — на 3.
3. **Near transfer (EDU.D.010) надёжен, far transfer — нет.** Перенос между доменами требует ЯВНОГО bridge.
4. **Meta-principles (EDU.FORM.004) — обязательны.** Retrieval practice и Spaced repetition — не опции, а элементы каждой единицы.
5. **Список принципов может расти.** При добавлении нового принципа (ZP.7+, FPF.*) конструктор добавляет ось в профиль с depth=0.

## Definition

Scaffolding is an instructional method in which the teacher provides
structured support to the learner performing a task that is beyond their
current independent capability but within reach with assistance (the Zone
of Proximal Development). As the learner gains competence, the support is
systematically reduced ("faded") until the learner can perform
independently.

The concept originates from Vygotsky's Zone of Proximal Development (ZPD)
and was formalized as "scaffolding" by Wood, Bruner & Ross (1976).

## Purpose

Enable the learner to accomplish tasks they cannot yet do alone, while
building the competence to eventually perform independently. Scaffolding
bridges the gap between current ability and target competency without
bypassing the learning process.

## Inputs

| Input | Description |
|-------|-------------|
| Learner's current level | Assessed baseline of what the learner can do independently |
| Target competency | The skill or understanding the learner should reach |
| Learning tasks | Sequenced tasks within the ZPD, calibrated to the learner |

## Outputs

| Output | Description |
|--------|-------------|
| Progression evidence | Observable data showing decreasing support needs over time |
| Independent performance | Learner demonstrates target competency without assistance |

## Roles Involved

- **Instructor (EDU.R.001):** Designs the scaffolding sequence, calibrates
  task difficulty, decides when to fade support.
- **Navigator (EDU.R.003):** Provides one-on-one scaffolding in real time,
  adjusts support moment-by-moment based on learner responses.

## Related Methods

- EDU.M.004 (Experiential Learning) — scaffolding can structure the
  experience phase of Kolb's cycle.
- EDU.M.002 (Problem-Based Learning) — facilitator guidance in PBL is a
  form of scaffolding.

## Key Distinctions

- **EDU.D.008 Adaptation ≠ Simplification.** Scaffolding adapts support to
  the learner's level — it does NOT simplify the target task. The task
  remains at full complexity; what changes is the amount and type of
  assistance. Simplification removes complexity; scaffolding manages it.

## Failure Modes

| ID | Failure Mode | Description |
|----|-------------|-------------|
| EDU.FM.001 | Premature removal of support | Support is faded before the learner has sufficient competence, leading to failure and demotivation |
| EDU.FM.002 | Permanent scaffolding | Support is never removed; the learner becomes dependent and does not develop independent capability |
| EDU.FM.003 | Misidentified ZPD | Tasks are set outside the Zone of Proximal Development — too easy (no learning) or too hard (scaffolding insufficient) |

## SoTA Status

**Status:** current
**Evidence strength:** Strong

**Basis:**
- Wood, D., Bruner, J.S., & Ross, G. (1976). "The role of tutoring in
  problem solving." *Journal of Child Psychology and Psychiatry*, 17(2),
  89-100.
- Vygotsky, L.S. (1978). *Mind in Society: The Development of Higher
  Psychological Processes.*
- Hattie, J. (2023). *Visible Learning: The Sequel.* d=0.82 for
  scaffolding (one of the highest effect sizes).
- Sweller, J. (2024). Expertise Reversal Effect: scaffolding that helps
  novices can HINDER experts who already have schemas. Must calibrate
  support to prior knowledge level (EDU.D.009).

**Notes:** Scaffolding is effective across all age groups, with particular
strength for children and beginners. It is one of the most empirically
supported instructional methods in education research. **Caveat:** For
experts, excessive scaffolding creates extraneous cognitive load (CLT)
— the Expertise Reversal Effect (Sweller, 2024). Calibrate scaffolding
level to prior knowledge (EDU.D.009), not to learning style.

**Human ↔ AI parallel (EDU.D.011):** Scaffolding ↔ Curriculum Learning
in ML: both present progressively harder tasks as competence grows.

## Definition

Problem-Based Learning (PBL) is an instructional method in which learners
encounter an ill-structured, real-world problem before receiving formal
instruction on the topic. Through investigation, learners identify their
own knowledge gaps, acquire necessary knowledge through self-directed
study, apply it to the problem, and reflect on the learning process.

PBL was developed by Howard Barrows at McMaster University Medical School
in the 1960s and formalized in Barrows (1986).

## Purpose

Develop problem-solving ability, self-directed learning skills, and deep
understanding by making the problem — not the lecture — the starting point
of learning. PBL integrates knowledge acquisition with application,
producing transferable competence rather than inert knowledge.

## Inputs

| Input | Description |
|-------|-------------|
| Ill-structured problem | A realistic problem without a single correct solution; requires investigation |
| Resource materials | Books, articles, databases, experts available for self-directed study |
| Facilitator guidance | A facilitator who guides the inquiry process without providing answers |

## Outputs

| Output | Description |
|--------|-------------|
| Solution | A reasoned, evidence-based response to the problem |
| Reflection on learning process | Explicit identification of what was learned, how, and what remains unknown |
| Identified knowledge gaps | List of what learners discovered they needed to learn |

## Roles Involved

- **Facilitator (EDU.R.002):** Guides the inquiry process, asks probing
  questions, ensures productive group dynamics. Does NOT provide answers
  or lecture.

## Related Methods

- EDU.M.001 (Scaffolding) — facilitator guidance in PBL is a form of
  scaffolding that fades as learners develop self-direction.
- EDU.M.003 (Case Method) — both use real-world situations; PBL is
  problem-first and self-directed, Case Method is analysis-centered with
  more instructor direction.
- EDU.M.004 (Experiential Learning) — PBL can serve as the "concrete
  experience" phase in Kolb's cycle.

## Key Distinctions

- **EDU.D.002 Teaching ≠ Facilitation.** In PBL the educator is a
  facilitator, not a teacher in the traditional sense. The facilitator
  does not transmit knowledge; they guide the learner's own inquiry
  process. Confusing these roles destroys PBL.
- **EDU.D.004 Knowledge ≠ Skill.** PBL targets both: knowledge is
  acquired in service of solving the problem (not for its own sake), and
  the problem-solving process itself builds skill. Assessing only
  knowledge misses half the outcome.

## Failure Modes

| ID | Failure Mode | Description |
|----|-------------|-------------|
| EDU.FM.004 | Problem too structured | The problem has a clear single answer and predetermined steps — no genuine inquiry occurs, PBL degrades to a worksheet |
| EDU.FM.005 | Absent facilitation | Facilitator is absent or passive; learners flounder without direction, frustration replaces learning |
| EDU.FM.006 | Theory-first contamination | Instructor lectures before presenting the problem, eliminating the knowledge-gap discovery that drives PBL |

## SoTA Status

**Status:** current
**Evidence strength:** Moderate-to-Strong (context-dependent)

**Basis:**
- Barrows, H.S. (1986). "A taxonomy of problem-based learning methods."
  *Medical Education*, 20(6), 481-486.
- Hmelo-Silver, C.E. (2004). "Problem-based learning: What and how do
  students learn?" *Educational Psychology Review*, 16(3), 235-266.
- Strobel & van Barneveld (2009). Meta-analysis: PBL superior for
  long-term retention and skill application; direct instruction superior
  for short-term factual recall.
- Hattie (2023): PBL d=0.35 (moderate) — but this average masks strong
  context-dependence: effective for applied domains, weaker for
  foundational knowledge.

**Notes:** PBL is most effective for adults and professionals (medicine,
engineering, law, business). It requires careful problem design and skilled
facilitation. Effectiveness is well-documented in medical education and
increasingly in engineering and management education.

**CLT consideration:** PBL can overload working memory for novices who
lack schemas to organize information (Sweller, 2006). Solution: combine
PBL with scaffolding (EDU.M.001) and worked examples for beginners.

**Three-Level placement (EDU.FORM.004):** Level 2 (Domain Method) —
effective for applied disciplines, must incorporate Level 1 meta-principles
(retrieval practice, feedback, spaced repetition).

## Definition

The Case Method is an instructional method in which learners analyze a
detailed description of a real situation (a "case") that involves a
decision point or dilemma. Learners must identify the key issues, analyze
the context, evaluate alternatives, and arrive at a reasoned decision or
recommendation. Discussion — typically facilitated in a group — is the
primary mechanism of learning.

The method originated at Harvard Law School in the 1870s (Christopher
Columbus Langdell) and was adopted by Harvard Business School in the 1920s.

## Purpose

Develop analytical judgment, contextual reasoning, and decision-making
skill by placing learners in the position of the decision-maker. The Case
Method bridges the gap between theoretical knowledge and practical
application by requiring learners to act on incomplete information in
realistic contexts.

## Inputs

| Input | Description |
|-------|-------------|
| Prepared case | A detailed, well-structured description of a real situation with sufficient data for analysis |
| Discussion questions | Guiding questions that direct analysis toward key issues and decision points |
| Facilitator guide | Teaching notes with analysis framework, expected insights, and facilitation plan |

## Outputs

| Output | Description |
|--------|-------------|
| Case analysis | Structured analysis of the situation: issues, stakeholders, constraints, alternatives |
| Decision rationale | Reasoned recommendation with explicit justification |
| Lessons learned | Generalizable principles extracted from the specific case |

## Roles Involved

- **Facilitator (EDU.R.002):** Leads the case discussion, manages group
  dynamics, ensures all perspectives are heard, synthesizes insights.
  This is the primary role in Case Method.
- **Instructor (EDU.R.001):** May provide contextual lectures or
  theoretical frameworks before or after the case discussion to anchor
  learning.

## Related Methods

- EDU.M.002 (Problem-Based Learning) — both use real-world situations.
  Case Method provides a complete narrative for analysis; PBL presents an
  open problem for investigation. Case Method is more structured.
- EDU.M.004 (Experiential Learning) — case analysis can serve as
  "reflective observation" on vicarious experience.
- EDU.M.005 (Backwards Design) — cases are often designed using backwards
  design to ensure alignment with learning objectives.

## Key Distinctions

- **EDU.D.004 Knowledge ≠ Skill.** The Case Method specifically develops
  analytical skill (the ability to reason about complex situations and make
  decisions), not just knowledge of facts or theories. Cases that test only
  recall of facts miss the point entirely.

## Failure Modes

| ID | Failure Mode | Description |
|----|-------------|-------------|
| EDU.FM.007 | Case as illustration | The case is used merely to illustrate a pre-taught concept; no genuine decision-making or analysis is required from learners |
| EDU.FM.008 | Discussion without framework | Discussion proceeds without an analytical framework; opinions are shared but no structured reasoning develops |
| EDU.FM.009 | Single-answer expectation | Facilitator steers toward one "correct" answer, shutting down the multi-perspective analysis that makes the Case Method effective |

## SoTA Status

**Status:** current
**Evidence strength:** Moderate-to-Strong (well-established in professional education)

**Basis:**
- Christensen, C.R. (1987). *Teaching and the Case Method.* Harvard
  Business School Press.
- Merseth, K.K. (1991). "The early history of case-based instruction:
  Insights for teacher education today." *Journal of Teacher Education*,
  42(4), 243-249.
- Nkhoma, M. et al. (2014). Meta-analysis of case method effectiveness
  in higher education.

**Notes:** The Case Method is most effective for managers, lawyers, medical
students, and other professionals who must make decisions under uncertainty.
It requires high-quality case materials and skilled facilitation. The
method is well-established and widely used in business, law, and medical
education worldwide.

**Three-Level placement (EDU.FORM.004):** Level 2 (Domain Method) —
highly effective for management, law, and medicine. Should incorporate
Level 1 meta-principles: retrieval (EDU.M.006) through preparation
questions, spacing (EDU.M.007) by distributing case discussions.

## Definition

Experiential Learning is an instructional method based on Kolb's (1984)
four-stage learning cycle:

1. **Concrete Experience (CE):** The learner engages in a direct experience
   or activity.
2. **Reflective Observation (RO):** The learner reflects on the experience
   from multiple perspectives.
3. **Abstract Conceptualization (AC):** The learner forms generalizations,
   principles, or theories from the reflection.
4. **Active Experimentation (AE):** The learner tests the new concepts in
   new situations, generating the next concrete experience.

The cycle is iterative: each round deepens understanding. Learning occurs
through the transformation of experience, not merely its accumulation.

## Purpose

Structure learning-by-doing so that experience is converted into
transferable knowledge and tested principles. Without the full cycle,
experience remains anecdotal and does not generalize. The method ensures
that action and reflection are both present and connected.

## Inputs

| Input | Description |
|-------|-------------|
| Experience opportunity | A situation where the learner can act, observe, and receive feedback |
| Reflection framework | Structured questions or prompts guiding reflective observation |
| Conceptual models | Theories or frameworks available for the abstract conceptualization stage |

## Outputs

| Output | Description |
|--------|-------------|
| Experiential learning journal | Written record of experiences, reflections, and emerging principles |
| Conceptual models | Generalizations and theories the learner has constructed from experience |
| Action plans | Plans for active experimentation to test new concepts |

## Roles Involved

- **Facilitator (EDU.R.002):** Designs the experience, provides reflection
  prompts, guides conceptualization, and creates conditions for
  experimentation.
- **Mentor (EDU.R.005):** Supports longer-term experiential learning
  cycles, helps the learner connect experiences across time, and provides
  perspective from their own experience.

## Related Methods

- EDU.M.001 (Scaffolding) — scaffolding can be applied within any stage
  of the experiential cycle, especially during abstract conceptualization.
- EDU.M.002 (Problem-Based Learning) — PBL can serve as the concrete
  experience that initiates the cycle.
- EDU.M.003 (Case Method) — case analysis is a form of reflective
  observation on vicarious experience.

## Key Distinctions

- **EDU.D.007 Teaching ≠ Informing.** Experiential learning is teaching
  through structured experience and reflection. Informing (transmitting
  facts) skips the experience and reflection stages entirely. A lecture
  about riding a bicycle is informing; riding, falling, reflecting, and
  riding again is experiential learning. The distinction determines
  whether the full cycle is activated.

## Failure Modes

| ID | Failure Mode | Description |
|----|-------------|-------------|
| EDU.FM.010 | Experience without reflection | Learners go through activities but skip reflective observation; experience does not convert to learning |
| EDU.FM.011 | Cycle truncated | One or more stages are skipped (commonly: reflection or experimentation); learning remains partial |
| EDU.FM.012 | Conceptualization without grounding | Abstract theories are introduced without prior concrete experience; concepts remain inert and disconnected |

## SoTA Status

**Status:** current
**Evidence strength:** Moderate (widely used, but effect sizes variable)

**Basis:**
- Kolb, D.A. (1984). *Experiential Learning: Experience as the Source
  of Learning and Development.* Prentice-Hall.
- Jarvis, P. (2006). *Towards a Comprehensive Theory of Human Learning.*
  Routledge.
- Moon, J.A. (2004). *A Handbook of Reflective and Experiential Learning.*
  Evidence for reflection as learning mechanism.

**Notes:** Experiential learning is effective across all age groups, with
particular strength for adults who bring prior experience to the cycle.
The model is widely applied in professional education, outdoor education,
service learning, and workplace training. Critics note the model's
linearity; Jarvis (2006) offers a more nuanced multi-path model.

**Enhancement with Level 1 meta-principles (EDU.FORM.004):** Embed
retrieval practice (EDU.M.006) in the Reflective Observation stage and
spaced repetition (EDU.M.007) between cycles for maximum effect.

**Three-Level placement:** Level 2 (Domain Method) — particularly
effective for interpersonal skills, leadership, and professional practice.

## Definition

Backwards Design (also known as Understanding by Design, UbD) is a
meta-method for instructional design developed by Wiggins & McTighe (2005).
Instead of starting with content or activities, the designer works
backwards from desired results through three stages:

1. **Stage 1 — Identify Desired Results:** What should learners understand
   and be able to do? Define learning goals, essential questions, and
   transfer objectives.
2. **Stage 2 — Determine Acceptable Evidence:** How will we know learners
   have achieved the results? Define assessment criteria, performance
   tasks, and evidence of understanding.
3. **Stage 3 — Plan Learning Experiences:** What activities, instruction,
   and resources will enable learners to achieve the results and produce
   the evidence? Design the learning sequence.

This is a meta-method: it is used to design instruction that may employ
other methods (scaffolding, PBL, case method, etc.) in Stage 3.

## Purpose

Ensure tight alignment between learning objectives, assessment, and
instruction. Backwards Design prevents the two most common design flaws:
(1) coverage-driven design (teaching content without clear outcomes) and
(2) activity-driven design (doing activities without clear assessment of
learning). By starting from the end, every element of the design serves
the learning goal.

## Inputs

| Input | Description |
|-------|-------------|
| Learning goals / standards | What learners should know, understand, and be able to do (external standards or internal objectives) |
| Context | Institutional constraints, time available, resources, prerequisites |
| Learner profile | Prior knowledge, motivations, challenges, and characteristics of the target learners |

## Outputs

| Output | Description |
|--------|-------------|
| Unit plan | A complete instructional design with aligned objectives, assessments, and activities |
| Assessment rubrics | Criteria for evaluating whether desired results have been achieved |
| Learning sequence | Ordered activities and experiences mapped to specific objectives and evidence |

## Roles Involved

- **Instructional Designer (EDU.R.004):** The primary user of Backwards
  Design. Conducts the three-stage design process, ensures alignment,
  and produces the unit plan.

## Related Methods

- EDU.M.001 (Scaffolding) — scaffolding sequences are designed in Stage 3
  of Backwards Design.
- EDU.M.002 (Problem-Based Learning) — PBL can be selected as the
  instructional strategy in Stage 3.
- EDU.M.003 (Case Method) — cases can be selected as learning experiences
  in Stage 3.
- EDU.M.004 (Experiential Learning) — experiential cycles can be planned
  in Stage 3.

## Key Distinctions

- **EDU.D.005 Content ≠ Methodology.** Backwards Design makes this
  distinction operational. Content (what is taught) is defined in Stage 1.
  Methodology (how it is taught) is defined in Stage 3. Stage 2
  (assessment) bridges them by specifying what evidence of learning looks
  like. Confusing content with methodology leads to "covering material"
  without achieving understanding.

## Failure Modes

| ID | Failure Mode | Description |
|----|-------------|-------------|
| EDU.FM.013 | Activity-first design | The designer starts with interesting activities (Stage 3) without first defining outcomes (Stage 1) and evidence (Stage 2); activities may be engaging but not aligned to learning goals |
| EDU.FM.014 | Assessment-objective mismatch | Assessment (Stage 2) does not actually measure the desired results from Stage 1; e.g., objectives call for transfer and application but assessment tests recall |
| EDU.FM.015 | Coverage mentality | Stage 1 is defined as "cover chapters 1-5" rather than as understanding and transfer goals; the design collapses back to content-driven instruction |

## SoTA Status

**Status:** current
**Evidence strength:** Strong (widely adopted, de facto standard)

**Basis:**
- Wiggins, G., & McTighe, J. (2005). *Understanding by Design* (2nd ed.).
  ASCD.
- Hattie (2023): Alignment of objectives-assessment-instruction is among
  the highest-impact factors (d=0.56 for "teacher clarity" which UbD
  ensures).
- Merrill, M.D. (2002). "First Principles of Instruction." Five
  principles that align with Backwards Design.

**Notes:** Backwards Design is the dominant framework for instructional
design in K-12 and higher education. It is a meta-method (used to design
instruction) rather than an instructional method itself. The framework is
compatible with and often used alongside all other methods in the EDU pack.
UbD has been adopted by curriculum standards organizations worldwide.

**Three-Level placement (EDU.FORM.004):** Meta-method — applies across
all three levels. Stage 1 uses Level 1 meta-principles, Stage 3 selects

## Definition

Retrieval Practice — метод обучения, при котором ученик активно
извлекает информацию из памяти (вспоминает), а не пассивно перечитывает
или переслушивает материал. Каждый акт успешного припоминания укрепляет
связи в памяти сильнее, чем повторное чтение того же материала.

Известен также как "testing effect" (эффект тестирования): тестирование
само по себе является обучением, а не только его измерением.

## Purpose

Перевести знания из пассивного «узнавания» (recognition) в активное
«припоминание» (recall), что обеспечивает значительно более прочное
и переносимое обучение.

## Inputs

| Input | Description |
|-------|-------------|
| Изученный материал | Содержание, которое ученик ранее встречал |
| Retrieval cues | Вопросы, подсказки, задания, требующие припоминания |
| Обратная связь | Правильный ответ после попытки припоминания |

## Outputs

| Output | Description |
|--------|-------------|
| Укреплённая память | Более прочное и долговременное запоминание |
| Выявленные пробелы | Области, где ученик не может вспомнить = зоны для повторного изучения |
| Метакогнитивная калибровка | Ученик точнее оценивает, что знает и чего не знает |

## Evidence Strength

**Strong.** Один из наиболее воспроизводимых эффектов в когнитивной науке.

| Источник | Результат |
|----------|-----------|
| Roediger & Karpicke (2006) | Testing effect: retrieval > re-reading для долговременного запоминания |
| Dunlosky et al. (2013) | Rated "high utility" — одна из 2 стратегий из 10 с высшей оценкой |
| Effect size | d = 0.50 (мета-анализ, Adesope et al., 2017) |
| Rowland (2014) | Мета-анализ 159 исследований: устойчивый эффект |

## Roles Involved

- **Instructor (EDU.R.001):** Проектирует retrieval-задания, обеспечивает
  обратную связь.
- **Instructional Designer (EDU.R.004):** Встраивает retrieval practice
  в структуру курса (quiz, flashcards, practice tests).

## Key Distinctions

- **EDU.D.009 Стиль обучения ≠ Уровень знаний.** Сложность retrieval
  заданий должна учитывать уровень предварительных знаний, а не
  «стиль обучения».
- **EDU.D.003 Формативное ≠ Суммативное.** Retrieval practice — это
  формативное оценивание: цель не в оценке, а в укреплении знаний.

## Synergy with Spaced Repetition

Retrieval Practice + Spaced Repetition (EDU.M.007) = максимальный эффект.
Извлечение из памяти через увеличивающиеся интервалы — «золотой стандарт»
запоминания в когнитивной науке.

## Implementation Patterns

| Паттерн | Описание | Когда использовать |
|---------|----------|--------------------|
| Low-stakes quiz | Короткий тест без оценки в начале занятия | Каждое занятие |
| Think-pair-share | Вспомни → обсуди с соседом → расскажи группе | Лекции, семинары |
| Flashcards | Карточки с вопросом и ответом (Anki, Quizlet) | Самостоятельное обучение |
| Brain dump | Написать всё, что помнишь по теме, за 5 мин | Начало занятия |
| Practice test | Полный тест без оценки, с обратной связью | Перед суммативным экзаменом |

## Failure Modes

| ID | Failure Mode | Description |
|----|-------------|-------------|
| FM.001 | Без обратной связи | Retrieval без feedback может закреплять ошибки |
| FM.002 | Слишком лёгкий retrieval | Узнавание (recognition) вместо припоминания (recall) — не даёт эффекта |
| FM.003 | Только фактическое | Retrieval только фактов (Level 1 Bloom) без retrieval на уровне применения/анализа |

## SoTA Status

**Status:** current

**Basis:**
- Roediger, H.L. & Karpicke, J.D. (2006). "Test-Enhanced Learning."
  *Psychological Science*, 17(3), 249-255.
- Dunlosky, J. et al. (2013). "Improving Students' Learning With Effective
  Learning Techniques." *Psychological Science in the Public Interest*, 14(1).
- Adesope, O. et al. (2017). Meta-analysis: d = 0.50 across 272 studies.

**Notes:** Retrieval practice — один из двух методов с рейтингом "high utility"

## Definition

Spaced Repetition (интервальное повторение) — метод обучения, при котором
повторение материала распределяется по увеличивающимся временным интервалам,
а не концентрируется в одной сессии ("cramming"). Основан на кривой забывания
Эббингауза: каждое повторение в момент, когда память начинает угасать,
значительно укрепляет долговременное запоминание.

## Purpose

Перевести информацию из кратковременной памяти в долговременную с минимальными
затратами времени. Оптимизировать расписание повторений по индивидуальной
кривой забывания ученика.

## Inputs

| Input | Description |
|-------|-------------|
| Учебный материал | Содержание, разбитое на дискретные единицы (карточки, вопросы, задачи) |
| График интервалов | Расписание повторений с увеличивающимися интервалами |
| Результат предыдущего повторения | Успешность припоминания определяет следующий интервал |

## Outputs

| Output | Description |
|--------|-------------|
| Долговременное запоминание | Материал удерживается месяцы и годы, не дни |
| Оптимизированное время | Тратится только время на повторение в критические моменты |
| Индивидуальная кривая | Данные о скорости забывания для каждой единицы |

## Evidence Strength

**Strong.** Один из наиболее воспроизводимых эффектов в когнитивной науке.

| Источник | Результат |
|----------|-----------|
| Cepeda et al. (2006) | Мета-анализ: distributed practice > massed practice (d=0.54) |
| Dunlosky et al. (2013) | Rated "high utility" — одна из 2 лучших стратегий из 10 |
| Ebbinghaus (1885) | Первооткрыватель кривой забывания |
| Pimsleur (1967) | Формализация graduated intervals для языкового обучения |
| Leitner (1972) | Система карточек с интервальным повторением |

## Spacing Schedule

Типичные интервалы (алгоритм SM-2, Wozniak 1987):

```
Первое повторение:    1 день
Второе повторение:    3 дня
Третье повторение:    7 дней
Четвёртое повторение: 14 дней
Пятое повторение:     30 дней
Далее:                ×2.5 от предыдущего интервала
```

При ошибке — интервал сбрасывается и начинается заново.

## Roles Involved

- **Instructional Designer (EDU.R.004):** Проектирует расписание курса
  с учётом spacing (не всё в один день).
- **Navigator (EDU.R.003):** Индивидуально калибрует интервалы под ученика.

## Key Distinctions

- **EDU.D.010 Ближний ≠ Дальний перенос.** Spaced repetition обеспечивает
  прочный ближний перенос. Для дальнего — недостаточен (нужны ещё
  interleaving и variability of practice).

## Human ↔ AI Parallel (EDU.D.011)

Кривая забывания Эббингауза ↔ catastrophic forgetting в нейросетях.
Spaced repetition ↔ rehearsal methods в continual learning.
Обе проблемы решают одну задачу: как удержать старое знание
при поступлении нового.

## Implementation Patterns

| Паттерн | Описание | Инструменты |
|---------|----------|-------------|
| SRS Software | Алгоритмические карточки с автоматическим расписанием | Anki, SuperMemo, Quizlet |
| Spiral Curriculum | Возврат к темам через увеличивающиеся интервалы в программе | Программы K-12 |
| Distributed Practice | Разнесение практических занятий по дням/неделям | Любой учебный план |
| Interleaved Review | Чередование старых и новых тем в каждом занятии | Семинары, уроки |

## Failure Modes

| ID | Failure Mode | Description |
|----|-------------|-------------|
| FM.001 | Cramming | Концентрация всего обучения перед экзаменом — кратковременный эффект, быстрое забывание |
| FM.002 | Fixed intervals | Одинаковые интервалы для всех учеников/всех карточек — неоптимально |
| FM.003 | Only recognition | Повторение = перечитывание (без retrieval) — минимальный эффект |

# EDU.M.008 — Spiral Curriculum (Спиральный учебный план)

## Определение

Метод организации учебной программы, при котором ключевые концепции (принципы мышления ZP + FPF) повторно изучаются через запланированные интервалы, каждый раз на возрастающей глубине (Bloom 0→5) и в новом контексте (доменная ротация).

## Три принципа Bruner (1960)

1. **Повторное посещение:** тема изучается многократно на протяжении всей программы
2. **Рост сложности:** каждое посещение — на более глубоком когнитивном уровне (Bloom)
3. **Связывание:** новое обучение ЯВНО связано со старым («Помнишь, как…? Теперь…»)

## Входы

| Вход | Описание | Источник |
|------|---------|---------|

| Выход | Описание |
|-------|---------|

## Алгоритм применения

```
1. Определи текущую глубину ученика по принципу (EDU.WP.004)
2. Выбери принцип для текущего витка (стратегия: breadth-first или bottleneck-first)
3. Определи Bloom-цель для следующей глубины (depth+1)
4. Выбери домен (правило R6: не повторяй домен предыдущего витка!)
5. Сформулируй bridge: «Помнишь, как мы [предыдущий виток]? Теперь…»
6. Спроектируй задание (Backwards Design: цель → оценка → активность)
7. Встрой retrieval practice: повторение ключевых идей предыдущих витков
8. Проведи занятие
9. Оцени: может ли ученик выполнить can-do утверждения глубины?
10. Обнови Learning Progression Map
```

## Три паттерна спирали

### Паттерн 1: Bloom-спираль (основной)

Каждый виток целит в следующую глубину Bloom:
```
Глубина 1 → Remember: узнай принцип
Глубина 2 → Understand: объясни своими словами
Глубина 3 → Apply: примени к новой ситуации
Глубина 4 → Analyze: найди нарушение
Глубина 5 → Create: создай систему на принципе
```

### Паттерн 2: CPA-спираль (Singapore Math)

Каждый виток меняет модальность представления:
```
Виток A → Concrete: физические объекты, манипуляции руками
Виток B → Pictorial: схемы, диаграммы, визуальные модели
Виток C → Abstract: формулы, символы, формальные определения
```

### Паттерн 3: Cosmic-спираль (Montessori)

Каждый виток расширяет контекст:
```
Виток 1 → Я: как принцип работает в моей жизни
Виток 2 → Семья/класс: как принцип работает в группе
Виток 3 → Общество: как принцип работает в городе/стране
Виток 4 → Мир: как принцип работает в природе/науке
Виток 5 → Универсальное: как принцип работает ВЕЗДЕ
```

## Типичные ошибки

1. **Повторение без углубления.** Если виток 2 = та же глубина Bloom, что и виток 1, это не спираль — это повторение. Каждый виток ОБЯЗАН быть глубже.
2. **Отсутствие bridge.** Без явной фразы-связки ученик не видит преемственности. «Это новая тема» — антипаттерн. Правильно: «Помнишь, как мы…?»
3. **Один домен на все витки.** Если ZP.1 изучается только на примерах из математики, ученик не увидит его транс-дисциплинарность. Правило R6: сменяй домен.
4. **Пропуск retrieval practice.** Без активного припоминания предыдущих витков знание не интегрируется. Начало каждого занятия = 2-минутный recall предыдущего.

## Кому подходит

Подходит ВСЕМ возрастам и глубинам. Spiral Curriculum — мета-метод, совместимый с любым другим методом из PACK-education (scaffolding, PBL, case method внутри витка). Форма витка адаптируется под когнитивные ограничения ученика (EDU.FORM.006 → таблица форм).

## Действие (7 шагов)

### Шаг 1. Прочитать профиль и состояние

**Вход:** ЦД (L1 + L2 + L3)
**Выход:** `context` — структура с полями: ступень, ИТ-уровень, состояние, энергия, стиль, домен, история

**Правила:**
- Если L3 не вычислен (нет calculation engine) → fallback на L1 самооценку
- Если история пуста (новый пользователь) → ступень = 0 (Случайный), все глубины = 0

### Шаг 2. Определить фазу программы

**Вход:** `context.ступень`
**Выход:** `phase` (1–4)

| Ступень | Фаза | Фокус |
|---------|------|-------|
| 0 Случайный | Ф1 Основания | Мотивация, мировоззрение, первые практики |
| 1 Практикующий | Ф1→Ф2 (переход) | Закрепление оснований + первые различения |
| 2 Систематический | Ф2→Ф3 | Различения + создание |
| 3 Дисциплинированный | Ф3→Ф4 | Создание + автономия |
| 4 Проактивный | Ф4 | Автономия, IWE, Pack, наставничество |

### Шаг 3. Выбрать направление

**Вход:** `phase`, матрица весов (§ Матрица), `context.история` (вчерашнее направление)
**Выход:** `direction` — одно из 6 направлений

**Алгоритм:**
1. Взять веса направлений для текущей фазы (§ Матрица)
2. **Корректировка по состоянию:**
   - Хаос → вес направления 6 (Организм) × 2.0, вес направления 2 (Навыки) × 0.5
   - Тупик → вес направления 3 (Ограничения) × 1.5, вес направления 1 (Картины мира) × 1.5
   - Поворот → без корректировки
   - Развитие → вес направления 4 (Экзокортекс) × 1.5
3. **Корректировка по энергии:** энергия ≤ 2 из 5 → вес направления 6 (Организм) × 1.5
4. **Ротация:** обнулить вес вчерашнего направления (R1 — не повторять 2 дня подряд)
5. Нормализовать веса → `доля_i = вес_i / сумма_весов`
6. Выбрать направление: weighted random или max(доля × gap)

### Шаг 4. Выбрать тему и глубину

**Вход:** `direction`, `context.глубины`, программа ЛР v2
**Выход:** `topic` + `bloom_depth`

**Алгоритм:**
1. Из программы (personal-development-v2) → список тем текущей фазы по выбранному направлению
2. Стратегия выбора:
   - **Bottleneck-first:** тема с max(gap × impact). Gap = разница между целевой и текущей глубиной
   - Если все глубины равны → round-robin (следующая по порядку программы)
3. `bloom_depth` = текущая глубина + 1
4. **Mastery-gate (FORM.007 R3):** Не повышать глубину, если can-do текущей не пройдены

### Шаг 5. Собрать контент

**Вход:** `topic`, `bloom_depth`, ячейки DS-principles-curriculum
**Выход:** `raw_content` — урок + вопросы + задание

**Алгоритм:**
1. Найти ячейку: `cells/{topic}@{bloom_depth}`
2. Если ячейка есть → извлечь: can-do, методы, задания, формы подачи, bridge
3. **Fallback (нет ячейки):** генерация из руководств через knowledge-mcp + LLM (Haiku)
4. Встроить retrieval practice (FORM.007 R4): recall по 1–2 предыдущим темам
5. Встроить bridge (FORM.007 R2): «Помнишь, как мы [предыдущая тема]? Теперь...»

### Шаг 6. Адаптировать

**Вход:** `raw_content`, `context` (стиль, домен, ИТ-уровень, ошибки)
**Выход:** `adapted_content` — финальное занятие

**Адаптации:**

| Параметр | Как адаптировать |
|----------|-----------------|
| **Домен примеров** | Профессиональный домен пользователя (из ЦД). Разработчик → примеры с кодом. Менеджер → примеры с проектами |
| **Стиль подачи** | Краткий / подробный / с аналогиями (из ЦД L1) |
| **Ошибки** | Если в истории есть ошибки по теме → усиленный retrieval, дополнительный вопрос |
| **ИТ-уровень** | См. § Scaffolding ИТ-навыков |

### Шаг 7. Упаковать

**Вход:** `adapted_content`, метаданные
**Выход:** Структурированный объект для бота

## Scaffolding ИТ-навыков

> **Принцип:** Портной не просто адаптирует формат — он **ведёт к IWE**. Каждое занятие приближает к самостоятельности с инструментами. Это scaffolding (EDU.M.001), встроенный в задания.

### Траектория по ИТ-уровням

| ИТ-уровень | Что в занятии | Куда ведёт | Примеры заданий |
|:---:|------------|-----------|-----------------|
| 0 | Всё через бот. Подводки: «Это можно сохранить в файл — скоро покажем как». Ссылки на обзорные видео «что такое VS Code / Git / CLI» | → ИТ=1 | «Посмотри 5-минутное видео о VS Code. Что понравилось?» |
| 1 | Бот + простые инструкции. Рекомендации изучить инструменты по видео/курсам. Первые шаги: «Открой VS Code, создай файл» | → ИТ=2 | «Установи VS Code по инструкции. Создай файл "мои-допущения.md"» |
| 2 | Задания с IWE: заметки в экзокортексе, тайм-трекер, черновики | → ИТ=3 | «Создай заметку в inbox, запиши 3 различения из занятия» |
| 3 | Задания через IWE: Claude Code, анализ Pack, автоматизация | Автономия | «Используй Claude Code для анализа своих допущений vs ZP.1» |

### Правила scaffolding

- **R-IT.1:** Каждое занятие содержит `it_scaffolding` — подводку к следующему уровню ИТ
- **R-IT.2:** Не перескакивать: ИТ=0 не получает заданий с VS Code. Только подводки и видео
- **R-IT.3:** Ссылки на внешние ресурсы (видео, курсы) = часть контента. Портной подбирает релевантные
- **R-IT.4:** Переход ИТ-уровня = событие в ЦД. Фиксируется при успешном выполнении задания уровня N+1
- **R-IT.5:** «Пропитаться» важнее «установить». Сначала — зачем (видео, примеры, истории), потом — как (инструкция)

- Bottleneck: Навыки (глубина=1, max gap). Вес=1.0, gap=max → **выбор: направление 2 (Навыки)**

**Шаг 4. Тема и глубина:**
- Программа ЛР v2 Ф1 → «Жизненное мастерство и степени» (§2.6)
- Текущая глубина = 1 (Remember) → целевая = 2 (Understand)
- Can-do глубины 1 пройдены ✓ → переход разрешён

**Шаг 5. Контент:**
- Ячейка `Навыки@2` — не готова → **fallback**: руководство «МС» §1.12
- Урок (~500 слов): метод = теория + инструмент. Рабочий продукт = результат метода. 4 степени мастерства
- Вопрос: «Какая разница между умением и навыком? Приведи пример из своей работы»
- Задание: «Выбери 1 метод из своей работы (например, code review). Определи свою степень мастерства»
- Retrieval: «В прошлый раз мы говорили об ограничениях. Назови одно ограничение, которое мешает мастерству»
- Bridge: «Помнишь, как мы разбирали, что такое мастерство? Теперь разберём, КАК оно устроено»

**Шаг 6. Адаптация:**
- Домен: backend → «Метод: TDD. Инструмент: pytest. РП: зелёные тесты. А какая у тебя степень в TDD?»
- Стиль: краткий → без лирических отступлений, факты + примеры
- it_scaffolding (ИТ=1): «Это задание можно выполнить в боте. Но если хочешь попробовать — создай файл `мастерство.md` в VS Code и запиши ответ туда. Вот 3-минутное видео, как создать файл в VS Code: [ссылка]»

# EDU.WP.005 — Diagnostic Assessment (Диагностический тест принципов)

## Назначение

Определить текущий **векторный профиль** (EDU.FORM.006) ученика: на какой глубине Bloom он владеет каждым принципом. Аналог IELTS для английского, но вместо 4 навыков (reading, writing, listening, speaking) — N принципов (ZP.1-6, FPF.*).

## Принципы диагностики

1. **Каждый принцип тестируется независимо.** Как IELTS тестирует reading и speaking отдельно, так и тест тестирует ZP.1 и ZP.3 отдельно.
2. **Адаптивность.** Тест начинает с середины (глубина 2-3) и адаптируется: если ответ правильный — глубже, если нет — проще. 3-5 вопросов на принцип достаточно.
3. **Can-do как рубрика.** Каждый вопрос проверяет конкретное can-do утверждение из матрицы ячеек (EDU.MAP.001).
4. **Результат = вектор, не скаляр.** Выход теста: `{ZP.1: 3, ZP.2: 1, ZP.3: 4, ZP.4: 2, ZP.5: 3, ZP.6: 2}` — НЕ «уровень 2.5».
5. **Без штрафов.** Тест выявляет текущее состояние, не оценивает. «Вот где ты сейчас» — не «вот твоя оценка».

## Структура теста

### Модуль 1: ZP-принципы (6 блоков)

Для каждого ZP-принципа — адаптивный блок из 3-5 вопросов:

```
Вопрос 1: Глубина 2 (Understand) — стартовый
  ├── Правильно → Вопрос 2: Глубина 3 (Apply)
  │                ├── Правильно → Вопрос 3: Глубина 4 (Analyze)
  │                │                ├── Правильно → Глубина 4 ✓ (при 2+ правильных на 4 → тест 5)
  │                │                └── Неправильно → Глубина 3 ✓
  │                └── Неправильно → Глубина 2 ✓
  └── Неправильно → Вопрос 2: Глубина 1 (Remember)
                     ├── Правильно → Глубина 1 ✓
                     └── Неправильно → Глубина 0 ✓
```

### Модуль 2: FPF-концепции (по готовности)

FPF тестируется только если mean(ZP) ≥ 2. Аналогичная адаптивная структура для каждого FPF-концепта.

### Модуль 3: Метакогнитивный профиль (опционально)

5 вопросов на метакогницию: рефлексия, самодиагностика, стратегия обучения. Не влияет на профиль принципов, но важен для выбора стратегии (breadth-first vs bottleneck-first).

## Типы вопросов (по глубине)

| Глубина | Bloom | Тип вопроса | Пример (ZP.1) |
|---------|-------|-------------|---------------|
| 1 | Remember | Выбор/распознавание | «Что из перечисленного является обоснованным утверждением?» (4 варианта) |
| 2 | Understand | Объяснение своими словами | «Объясни разницу между фактом, мнением и гипотезой» (открытый ответ) |
| 3 | Apply | Задача в новом контексте | «Вот утверждение из новостей. Найди исходную посылку и построй цепочку» |
| 4 | Analyze | Анализ и критика | «Вот аргумент двух сторон спора. Найди скрытую аксиому каждой стороны» |
| 5 | Create | Создание | «Сформулируй 3 аксиомы для области X. Выведи 2 следствия из каждой» |

## Формы проведения

| Форма | Длительность | Кто проводит | Когда применять |
|-------|-------------|-------------|----------------|
| **Бот-тест** (будущее) | 15-30 мин | Адаптивный бот (implementation) | Первичная диагностика, массовый охват |
| **Self-assessment** | 10-15 мин | Сам ученик (чек-лист can-do) | Быстрая самодиагностика, мотивация |
| **Очное интервью** | 30-60 мин | Навигатор / Диагност | Глубокая диагностика, начало программы |
| **Портфолио-ревью** | 1-2 ч | Instructional Designer | Опытные ученики, переход на bottleneck-first |

## Выход: Начальный профиль

```yaml
diagnostic_result:
  date: 2026-09-01
  learner: "Иван"
  method: bot_test  # bot_test | self_assessment | interview | portfolio

  profile:
    ZP.1_axiomaticity: 3
    ZP.2_structure: 2
    ZP.3_multiscale: 2
    ZP.4_optimization: 1
    ZP.5_probability: 3
    ZP.6_limits: 1
    FPF.holon: 1
    FPF.bounded_context: 0
    FPF.trust: 0

  summary:
    mean_zp: 2.0
    min_zp: 1
    bottleneck: ZP.4  # gap=2, impact=3 → priority=6
    strategy: breadth-first  # min_zp < 2 → ещё рано для bottleneck-first

  recommendation: "Стратегия breadth-first. Фокус: ZP.4 и ZP.6 на глубину 2 (самые слабые). Параллельно все ZP поднять до min=2."
```

## Критерии существования

1. Тест содержит ≥3 вопроса на каждый ZP-принцип
2. Каждый вопрос привязан к can-do из EDU.MAP.001
3. Адаптивная логика определена (правило перехода между глубинами)
4. Результат = векторный профиль (не скалярный «балл»)
5. Есть рекомендация стратегии и bottleneck

# EDU.FM.001 — Лекция = Обучение

## Definition

Подмена реального обучения передачей информации: преподаватель прочитал лекцию — значит, обучение состоялось. Смешивается акт информирования (однонаправленная передача данных) с обучением (процесс, приводящий к изменению способностей ученика). Лекция может быть частью обучения, но сама по себе обучением не является.

## Category

**distinction-confusion** — нарушение различения EDU.D.007 (Обучение != Информирование). Преподаватель путает передачу информации с формированием навыков и понимания.

## Observable Symptoms

| Симптом | Как проявляется |
|---------|----------------|
| Неспособность применить материал | Студенты слушали лекцию, но не могут решить задачу по теме без конспекта |
| Стабильно низкие показатели | Pass/fail rates не улучшаются от семестра к семестру несмотря на «качественные» лекции |
| Разрыв «понимаю — не могу» | Студенты отчитываются «всё понятно», но проваливают практические задания |
| Пассивная аудитория | На лекции нет вопросов, нет взаимодействия — только запись/слушание |
| Оценка преподавателя по подаче | «Хороший лектор» = интересно рассказывает, а не «ученики научились» |

## Root Causes

1. **Нарушение различения EDU.D.007 (Обучение != Информирование)** — ключевая причина. Преподаватель не различает передачу информации и процесс обучения
2. **Историческая традиция** — лекционный формат доминирует в высшем образовании с XII века (средневековый университет), закреплён инерцией системы
3. **Экономика масштаба** — лекция «дёшева»: один преподаватель на 200 студентов. Активное обучение требует малых групп
4. **Отсутствие обратной связи** — без формативного оценивания (EDU.D.003) преподаватель не видит, что обучения не происходит
5. **Подмена метрики** — успех измеряется «покрытием материала» (что рассказал), а не «изменением способностей» (что ученик теперь может)

## Consequences

| Последствие | Описание |
|-------------|----------|
| Иллюзия обучения | Преподаватель считает, что научил; студент считает, что понял — оба ошибаются |
| Потерянное время | Часы лекций без результата — невосполнимый ресурс |
| Самообвинение студентов | «Не могу — значит я глупый», хотя причина — метод обучения, а не способности |
| Воспроизводство неэффективности | Студент, ставший преподавателем, копирует лекционную модель |
| Разочарование в предмете | «Предмет скучный/сложный» — хотя проблема не в предмете, а в методе |

## Detection Methods

1. **Тест применения:** Попросить студентов решить задачу по теме без конспекта сразу после лекции. Если >50% не справляются — лекция не равна обучению
2. **Анализ времени:** Подсчитать долю активного времени студентов (говорят, решают, обсуждают) vs пассивного (слушают, записывают). Если пассивное >80% — информирование, не обучение
3. **Отложенный тест:** Через 1 неделю после лекции дать тест без предупреждения. Retention <30% — лекция не создала знания
4. **Аудит обратной связи:** Есть ли в лекции точки проверки понимания (формативное оценивание)? Если нет — нет механизма обнаружения проблемы

# EDU.FM.002 — Один размер для всех

## Definition

Применение одного и того же метода обучения, темпа и материала ко всем участникам группы без учёта возрастных, профессиональных и уровневых различий. Преподаватель выбирает «средний» подход, который не оптимален ни для одной подгруппы.

## Category

**method-failure** — ошибка в выборе и применении метода обучения. Один метод применяется к аудитории, требующей дифференцированного подхода.

## Observable Symptoms

| Симптом | Как проявляется |
|---------|----------------|
| Высокий dropout в смешанных группах | Отсев непропорционально высок — уходят и «сильные» (скучно), и «слабые» (непонятно) |
| Бимодальное распределение | Оценки удовлетворённости показывают два пика (очень довольны и очень недовольны), а не нормальное распределение |
| Скучающие продвинутые | Опытные участники отвлекаются, уходят с занятий раньше, дают обратную связь «слишком базово» |
| Потерянные начинающие | Новички не успевают за темпом, перестают задавать вопросы, тихо «выпадают» |
| Универсальный учебный план | Одна и та же программа для всех потоков без вариаций |
| Усреднённые метрики | Средний балл удовлетворённости 3.5/5 скрывает разрыв между 4.8 и 2.1 |

## Root Causes

1. **Нарушение различения EDU.D.001 (Педагогика != Андрагогика)** — методы для детей применяются к взрослым или наоборот, без учёта фундаментальных различий в мотивации и роли опыта
2. **Нарушение различения EDU.D.008 (Адаптация != Упрощение)** — вместо адаптации метода под аудиторию преподаватель либо упрощает для всех, либо не меняет ничего
3. **Экономия усилий** — создание одной программы вместо нескольких вариантов дешевле в подготовке, но дороже в результатах
4. **Отсутствие диагностики входного уровня** — преподаватель не знает состав аудитории до начала обучения
5. **Институциональное ограничение** — учебное заведение требует единый учебный план, не предоставляя инструментов дифференциации

## Consequences

| Последствие | Описание |
|-------------|----------|
| Субоптимальное обучение для всех | Ни одна подгруппа не получает метод, оптимальный для неё |
| Потеря лучших и слабейших | Уходят наиболее мотивированные (скучно) и наименее подготовленные (сложно) — остаётся только «середина» |
| Иллюзия успешности | Средние показатели выглядят «нормально», скрывая провал на краях распределения |
| Неэффективное использование ресурсов | Время и бюджет тратятся на обучение, не создающее максимальной ценности |
| Демотивация преподавателя | Постоянная обратная связь «слишком сложно / слишком просто» без возможности угодить всем |

3. **Анализ dropout:** Сравнить профили ушедших. Если уходят и «сильные», и «слабые» — один метод не работает для разных подгрупп
4. **Аудит учебного плана:** Проверить: есть ли в программе вариативные треки, уровни сложности, адаптивные задания? Если нет — «один размер для всех»
5. **Сегментированная обратная связь:** Собирать feedback не средним баллом, а с разбивкой по подгруппам (уровень, возраст, профессия). Разрыв >1.5 балла между подгруппами = проблема

# EDU.FM.003 — Проверка знаний вместо навыков

## Definition

Система оценивания проверяет способность запомнить и воспроизвести информацию (уровни 1-2 по таксономии Блума: запоминание, понимание), но не способность применить, проанализировать, оценить или создать (уровни 3-6). Экзамен измеряет не то, что должен.

## Category

**work-product-failure** — ошибка в рабочем продукте (системе оценивания). Артефакт оценки не соответствует заявленным целям обучения.

## Observable Symptoms

| Симптом | Как проявляется |
|---------|----------------|
| Зубрёжка перед экзаменом | Студенты интенсивно запоминают за 1-2 дня до экзамена и забывают через неделю |
| Высокие баллы, низкая компетенция | Отличники не могут решить задачу из реальной практики по той же теме |
| Однотипные вопросы | Экзамен состоит из тестов с выбором ответа, определений, пересказа — всё уровни 1-2 |
| Оценка на уровнях 1-2 Блума | В заданиях доминируют глаголы: «перечислите», «назовите», «дайте определение» |
| Недовольство работодателей | Выпускники «знают теорию», но не могут работать — разрыв между дипломом и навыком |
| Шпаргалки как стратегия | Студенты делают шпаргалки с фактами, а не с методами решения — потому что экзамен это и проверяет |

## Root Causes

1. **Нарушение различения EDU.D.004 (Знание != Навык)** — преподаватель не различает декларативное знание (знаю что) и процедурное (могу сделать)
2. **Нарушение различения EDU.D.003 (Формативное != Суммативное оценивание)** — суммативное оценивание проверяет «запомнил», а не «научился»
3. **Простота создания** — написать 50 вопросов на запоминание проще, чем создать 10 задач на применение. Проверять тесты проще, чем оценивать проекты
4. **Масштабируемость** — тест с выбором ответа можно проверить автоматически на 500 студентов; оценка навыка требует эксперта
5. **Традиция** — «экзамен = билеты = ответ по памяти» закреплён в образовательной культуре
6. **Отсутствие модели компетенций** — не определено, какие навыки (а не знания) должен продемонстрировать выпускник

## Consequences

| Последствие | Описание |
|-------------|----------|
| Ложноположительный результат | Система оценивания сообщает «обучен», хотя навык не сформирован |
| Выпускники без компетенций | Диплом не гарантирует способность выполнять работу |
| Недовольство работодателей | Разрыв между ожиданиями рынка и результатами образования |
| Деградация мотивации | Студент учится «сдавать экзамен», а не «уметь делать» — подмена цели |
| Невалидная обратная связь | Преподаватель не получает информации о реальных пробелах в обучении |

1. **Аудит по таксономии Блума:** Классифицировать каждое задание экзамена по уровням Блума. Если >60% заданий на уровнях 1-2 (запоминание, понимание) — тест проверяет знания, не навыки
2. **Тест переноса:** Дать студентам задачу из реальной практики (не из учебника). Если успешные на экзамене студенты не справляются — экзамен невалиден
3. **Retention test:** Провести тот же экзамен через 3 месяца без предупреждения. Если результаты падают >50% — было запомнено, а не усвоено
4. **Анализ глаголов:** Проверить формулировки заданий. Преобладание «назовите», «определите», «перечислите» vs «решите», «создайте», «оцените» — индикатор уровня проверки
5. **Обратная связь от работодателей:** Систематический опрос работодателей выпускников: «Какие навыки отсутствуют?» Повторяющиеся ответы = пробелы в оценивании

# EDU.FM.004 — Контент без практики

## Definition

Перегрузка обучения теоретическим материалом без предоставления возможности применить знания на практике. Преподаватель стремится «покрыть» весь материал, жертвуя временем на упражнения, задачи и эксперименты. Результат — информационное ожирение без формирования навыков.

## Category

**method-failure** — ошибка в методе обучения. Баланс теория/практика нарушен в пользу теории, что делает обучение неэффективным.

## Observable Symptoms

| Симптом | Как проявляется |
|---------|----------------|
| Информационная перегрузка | Студенты отмечают: «слишком много материала», «не успеваю переварить» |
| Низкий retention | Через 1 неделю после занятия студенты помнят <20% материала |
| Нет изменения поведения | После курса участники делают то же, что и до него — знания не конвертировались в действия |
| «Интересно, но бесполезно» | Характерная обратная связь: контент оценивается высоко, но полезность — низко |
| Конспектомания | Студенты судорожно записывают всё, потому что не могут обработать в реальном времени |
| Плотные слайды | Презентация содержит 80+ слайдов с текстом; каждый слайд — информация без задачи |

## Root Causes

1. **Проклятие знания (Expert Blind Spot)** — эксперт не помнит, каково это — не знать предмет. Ему всё кажется «базовым» и «быстро объясняемым»
2. **Давление программы** — «нужно успеть пройти весь материал за семестр» — покрытие важнее усвоения
3. **Смешение покрытия с обучением** — «я всё рассказал = я всё преподал». Нарушение EDU.D.007 (Обучение != Информирование)
4. **Нарушение различения EDU.D.004 (Знание != Навык)** — теория создаёт знание (уровни 1-2), но навык (уровни 3-6) требует практики
5. **Страх «пустого» времени** — молчание, работа в группах, самостоятельное решение кажутся преподавателю «потерей времени»
6. **Отсутствие дидактического дизайна** — программа строится от содержания («что рассказать»), а не от результата («что ученик сможет делать»)

## Consequences

| Последствие | Описание |
|-------------|----------|
| Информационное ожирение | Много информации — мало знаний. Данные не превращаются в способности |
| Выученная беспомощность | Студент привыкает: «мне объясняют, я не понимаю, значит я не способный» |
| Отчуждение от предмета | Перегрузка вызывает защитную реакцию — отторжение материала и предмета целиком |
| Невозможность обнаружить пробелы | Без практики ни студент, ни преподаватель не знают, что именно не усвоено |
| Потерянные инвестиции | Время и деньги на обучение не конвертируются в результат |

## Detection Methods

1. **Ratio теория/практика:** Подсчитать долю времени на теорию vs практику в программе. Если теория >70% — вероятен данный failure mode
2. **Тест retention:** Через 7 дней после занятия провести незапланированный тест. Retention <30% — материал не усвоен (только прослушан)
3. **Анализ программы от результата:** Для каждого блока программы задать вопрос: «Какую задачу студент сможет решить после этого блока?» Если ответа нет — блок информирует, но не обучает
4. **Подсчёт слайдов на единицу времени:** >2 информационных слайдов на минуту = информационный поток, не обучение
5. **Обратная связь «через месяц»:** Спросить участников через 30 дней: «Что вы используете из курса?» Если ответ «ничего» или «не помню» — контент не конвертировался в практику

# EDU.FM.005 — Игнорирование предварительного опыта

## Definition

Обучение взрослых людей ведётся так, будто они — чистый лист (tabula rasa): без учёта их профессионального опыта, существующих знаний, сформированных моделей и практических навыков. Преподаватель применяет педагогический подход (для детей) к аудитории, чей опыт — главный ресурс обучения.

## Category

**distinction-confusion** — нарушение различения EDU.D.001 (Педагогика != Андрагогика). Преподаватель не различает фундаментально разные подходы к обучению детей и взрослых.

## Observable Symptoms

| Симптом | Как проявляется |
|---------|----------------|
| Отчуждение взрослых учеников | Опытные профессионалы отключаются, смотрят в телефон, перестают участвовать |
| «Я это уже знаю» | Повторяющиеся жалобы на базовость материала от участников с опытом |
| Сопротивление «школьным» активностям | Взрослые отказываются от заданий, которые воспринимают как унизительные или инфантильные |
| Экспертиза в зале не используется | У участников 10-20 лет опыта, но преподаватель — единственный источник знаний |
| Примеры не из жизни аудитории | Кейсы и примеры абстрактные или из другой отрасли, не связанные с реальностью участников |
| Высокий dropout опытных | Наиболее квалифицированные участники уходят первыми — им нечего брать |

## Root Causes

1. **Нарушение различения EDU.D.001 (Педагогика != Андрагогика)** — преподаватель не изучал андрагогику или не принимает её как отдельную дисциплину
2. **Нарушение различения EDU.D.006 (Внешняя != Внутренняя мотивация)** — для взрослых внутренняя мотивация (связь с реальными задачами) критична, но используются внешние стимулы (оценки, сертификаты)
3. **Преподаватель не знает аудиторию** — не проводит предварительную диагностику опыта и потребностей участников
4. **Институциональная инерция** — учебное заведение строит программы по педагогической модели даже для MBA и корпоративного обучения
5. **Страх потери контроля** — если признать, что участники знают больше преподавателя в своей области, роль «эксперта» оказывается под угрозой
6. **Шаблонный подход** — одна и та же программа используется для стажёров и директоров без адаптации

## Consequences

| Последствие | Описание |
|-------------|----------|
| Потеря вовлечённости ценных учеников | Наиболее опытные и мотивированные участники — первые, кто уходит |
| Неиспользованная экспертиза | В аудитории из 20 человек может быть 200 лет совокупного опыта, который не вовлекается в процесс |
| Dropout | Взрослые голосуют ногами: не могут заставить себя «учиться как в школе» |
| Снижение репутации | Программа получает отзывы «не для практиков», «слишком академично», «оторвано от реальности» |
| Упущенная синергия | Обмен опытом между участниками мог бы создать ценность, превышающую вклад преподавателя |

## Detection Methods

1. **Аудит учебного плана на использование опыта:** Проверить программу: включает ли она активности, опирающиеся на опыт участников (case study из их практики, обмен опытом, peer review)? Если нет — данный failure mode
2. **Предварительная диагностика:** Проводится ли перед обучением опрос опыта, потребностей и ожиданий? Если нет — преподаватель не может учесть опыт, даже если хочет
3. **Анализ ролей:** Кто говорит >70% времени? Если преподаватель — опыт аудитории не используется. В андрагогике соотношение должно быть ближе к 40/60 (преподаватель/участники)
4. **Обратная связь по релевантности:** Спросить участников: «Насколько примеры и задания связаны с вашей реальной работой?» Оценка <3/5 — отрыв от опыта аудитории
5. **Тест адаптации:** Изменил ли преподаватель программу после получения информации о составе группы? Если программа идентична для всех групп — опыт игнорируется системно

# EDU.FM.006 — Поверхностная геймификация

## Definition

Добавление игровых элементов (баллы, бейджи, рейтинги, progress bars) поверх
традиционной дидактики без изменения самого процесса обучения. Создаёт иллюзию
вовлечённости: ученик набирает баллы, но не учится. Подменяет внутреннюю
мотивацию к обучению (EDU.D.006) внешними стимулами, которые исчезают вместе
с игровым контекстом.

## Category

**method-misapplication** — неверное применение метода (геймификация)
без понимания его ограничений. Нарушает EDU.D.006 (Внешняя ≠ Внутренняя мотивация).

## Observable Symptoms

| Симптом | Как проявляется |
|---------|----------------|
| Высокая "вовлечённость", низкие результаты | Ученики проводят много времени в системе, но не демонстрируют навыков |
| Гонка за баллами | Ученики оптимизируют набор баллов, а не обучение (gaming the system) |
| Зависимость от наград | Без баллов/бейджей мотивация падает до нуля |
| Поверхностные действия | Ученики кликают, отмечают, прогрессируют — но не думают |
| Разочарование после завершения | Нет внутреннего удовлетворения, только усталость от «игры» |

## Root Causes

1. **Нарушение EDU.D.006** — подмена внутренней мотивации внешней.
   Баллы и бейджи — внешние стимулы, не формирующие интерес к предмету
2. **Маркетинговое давление** — «геймификация» как модное слово,
   обещающее вовлечённость без усилий по изменению дидактики
3. **Путаница вовлечённости и обучения** — ученик кликает = ученик учится (ложь)
4. **Отсутствие desirable difficulties** — геймификация часто упрощает задачи
   (чтобы игрок "побеждал"), что снижает обучение (Bjork, 1994)

## Evidence Strength

**Moderate.** Мета-анализы геймификации показывают смешанные результаты.

| Источник | Результат |
|----------|-----------|
| Hamari et al. (2014) | Систематический обзор: позитивные эффекты на вовлечённость, но НЕ на обучение |
| Dicheva et al. (2015) | Обзор 34 исследований: баллы/бейджи без изменения дидактики — нет эффекта |
| Deci et al. (1999) | Мета-анализ SDT: внешние награды подавляют внутреннюю мотивацию |
| Toda et al. (2018) | Фреймворк: различает "structural gamification" (поверхностная) vs "content gamification" (интегрированная) |

**Различение**: Content gamification (интеграция игровой механики в учебное задание) —
работает. Structural gamification (баллы поверх обычного курса) — не работает.

## Consequences

| Последствие | Описание |
|-------------|----------|
| Подавление внутренней мотивации | Внешние награды вытесняют собственный интерес ученика (overjustification effect) |
| Ложная метрика | "Completion rate 95%" при нулевом обучении — обманывает всех |
| Ресурсы не туда | Бюджет на геймификацию вместо redesign дидактики |
| Краткосрочность | Эффект новизны: 2-3 недели вовлечённости, затем спад |

## Detection Methods

1. **Тест результатов:** Сравнить обучение с геймификацией и без.
   Если разницы нет — геймификация поверхностная
2. **Тест без наград:** Убрать все игровые элементы.
   Если мотивация исчезает — была только внешняя
3. **Анализ действий:** Что делает ученик для набора баллов?
   Если можно набрать баллы без обучения — система сломана
4. **Desirable difficulty test:** Становятся ли задания сложнее?
   Если всегда легко "выигрывать" — нет обучения

# EDU.MAP.001 — Шаблон ячейки Компетенция × Глубина

## Назначение

Абстрактный шаблон для описания того, ЧТО значит владеть любой компетенцией на каждой глубине Bloom. Применим к любому домену — от нулевых принципов (ZP) до профессиональных навыков.

**Pack EDU определяет шаблон. DS-программы заполняют его конкретным содержанием.**

```

      ↓ применяется к конкретным компетенциям
```

## Шаблон ячейки

Каждая ячейка (Компетенция C, Глубина D) описывается по стандартному шаблону:

```yaml
cell:
  principle: <ID>              # Идентификатор компетенции (ZP.1, FPF.Holon, SKILL.X, ...)
  principle_name: <название>
  depth: <1-5>                 # Bloom depth
  bloom_level: <Remember|Understand|Apply|Analyze|Create>

  # 1. Что может делать (can-do утверждения)
  can_do:
    - "<утверждение 1>"        # Наблюдаемое поведение (начинается с «Могу...»)
    - "<утверждение 2>"        # Каждое — проверяемое за <5 мин
    - "<утверждение 3>"        # 3-5 штук на ячейку

  # 2. Методы обучения
  methods:
    primary: [<EDU.M.xxx>]     # Основные методы для этой глубины
    meta: [EDU.M.006, EDU.M.007]  # Retrieval + Spaced (ВСЕГДА на каждой глубине)
    notes: "<специфика для данной компетенции на данной глубине>"

  # 3. Пререквизиты
  prerequisites:
    same_principle: <C@(D-1)>  # Предыдущая глубина той же компетенции
    cross_principle: [<список>] # Другие компетенции, которые нужны
    cognitive: "<минимальный когнитивный уровень>"

  # 4. Оценка (как проверить)
  assessment:
    type: <observation|artifact|test|portfolio|peer_teaching>
    criteria: "<что именно наблюдаем/проверяем>"
    transfer_test: "<задание в НОВОМ домене для проверки переноса>"
    retrieval_test: "<вопрос для проверки через 1+ неделю>"

  # 5. Типичные ошибки на этой глубине
  common_errors:
    - "<ошибка 1>: <почему возникает>"
    - "<ошибка 2>: <почему возникает>"

  # 6. Домены для обучения (ротация)
  domains: [<домен1>, <домен2>, <домен3>]  # Не менее 3, отличаются от D-1

  # 7. Формы подачи (по когнитивному уровню)
  forms:
    preoperational: "<форма для дооперационального уровня>"
    concrete_operational: "<форма для конкретно-операционального>"
    formal_operational: "<форма для формально-операционального>"
    postformal: "<форма для постформального>"

  # 8. Bridge (связь с предыдущей глубиной)
  bridge_template: "Помнишь, как мы <что делали на D-1>? Теперь <что делаем на D>."
```

## Правила заполнения

### Can-do утверждения
- Начинаются с «Могу...» (наблюдаемое действие)
- НЕ «знает», «понимает», «осознаёт» — это ненаблюдаемо
- Каждое проверяемо за <5 мин (в тесте или наблюдении)

### Методы
- ВСЕГДА включать EDU.M.006 (Retrieval) и EDU.M.007 (Spaced) — мета-принципы Level 1
- Primary — выбирать по таблице ниже (глубина → методы)

### Пререквизиты
- `same_principle`: всегда C@(D-1), кроме D=1 (null)
- `cross_principle`: только если ДЕЙСТВИТЕЛЬНО блокирует

### Assessment
- `transfer_test` ОБЯЗАТЕЛЕН — без него неизвестно, произошёл ли перенос (EDU.D.010)
- `retrieval_test` ОБЯЗАТЕЛЕН — без него неизвестно, прочно ли усвоение

### Домены
- Минимум 3 для ротации
- ОТЛИЧАЮТСЯ от предыдущей глубины

### Формы
- Для каждого когнитивного уровня — конкретная активность с длительностью
- Если глубина слишком высока для уровня — «Не применимо»

| Глубина | Bloom | Рекомендуемые методы | Assessment type |
|---------|-------|---------------------|-----------------|

## Cell Quality Rubric (CQR)

Содержательная оценка качества заполненной ячейки. 6 измерений × 3 уровня (0/1/2).

**Gate:** общий балл ≥ 9/12, ни одного нуля.

| # | Измерение | 0 (fail) | 1 (weak) | 2 (strong) |
|---|-----------|----------|----------|------------|
| 1 | **Специфичность can-do** | Ненаблюдаемое («знает», «понимает») | Наблюдаемое, но широкое | Наблюдаемое + тестируемое за <5 мин |
| 2 | **Обоснование метода** | Метод без notes | Notes есть, но generic | Notes объясняют ПОЧЕМУ этот метод для ЭТОГО принципа на ЭТОЙ глубине |
| 3 | **Валидность переноса** | Нет transfer_test | Transfer в смежном домене | Transfer в принципиально новом домене |
| 4 | **Покрытие ошибок** | Нет ошибок | Generic ошибки | Конкретные когнитивные паттерны + ПОЧЕМУ возникают |
| 5 | **Конкретность форм** | «Обсуждение» | Активность + длительность | Активность + длительность + конкретные материалы/вопросы |
| 6 | **Когерентность bridge** | Нет (кроме D=1) | Generic связка | Явно связывает навык D-1 → действие D |

**Применение:** self-check (v1), автоматическая валидация (v2), калибровка по данным учеников (v3).
